# EXPLAIN 参考结果 - bad.sql（LIKE 前导通配符致索引失效）

## MySQL 8.0（20 万行数据）

```
+----+-------------+----------------+------------+------+---------------+------+---------+------+--------+----------+-------------+
| id | select_type | table          | partitions | type | possible_keys | key  | key_len | ref  |   rows | filtered | Extra       |
+----+-------------+----------------+------------+------+---------------+------+---------+------+--------+----------+-------------+
|  1 | SIMPLE      | t_user_search  | NULL       | ALL  | NULL          | NULL | NULL    | NULL | 199,687 |   100.00 | Using where |
+----+-------------+----------------+------------+------+---------------+------+---------+------+--------+----------+-------------+
```

## 关键问题

| 字段 | 值 | 分析 |
|------|-----|------|
| type | `ALL` | 全表扫描 |
| possible_keys | `NULL` | 前导 % 导致索引无法使用 |
| key | `NULL` | idx_username 未被使用 |
| rows | ~199,687 | 扫描全部 20 万行 |
| Extra | `Using where` | 逐行检查 username 是否包含 'zhang' |

## 为什么慢

B+ 树索引按 `username` 有序排列。`LIKE '%zhang%'` 的前导通配符 `%` 意味着匹配可以出现在任意位置，B+ 树无法确定扫描起点（不像 `LIKE 'zhang%'` 可以定位到以 `zhang` 开头的位置）。

优化器只能放弃索引，全表扫描 20 万行，对每行的 `username` 做子串匹配。

实际耗时约 80ms。匹配到 5 行（zhang_san, zhang_si, zhang_wu, li_zhang, wang_zhang）。

## MySQL 5.7 差异

5.7 与 8.0 行为一致，均为全表扫描。
