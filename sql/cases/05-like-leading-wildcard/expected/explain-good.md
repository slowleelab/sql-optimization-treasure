# EXPLAIN 参考结果 - good.sql（后导通配符走索引范围扫描）

## MySQL 8.0（20 万行数据）

```
+----+-------------+---------------+------------+-------+---------------+--------------+---------+------+------+----------+-------------+
| id | select_type | table         | partitions | type  | possible_keys | key          | key_len | ref  | rows | filtered | Extra       |
+----+-------------+---------------+------------+-------+---------------+--------------+---------+------+------+----------+-------------+
|  1 | SIMPLE      | t_user_search | NULL       | range | idx_username  | idx_username | 202     | NULL |    3 |   100.00 | Using where |
+----+-------------+---------------+------------+-------+---------------+--------------+---------+------+------+----------+-------------+
```

## 关键改进

| 字段 | 值 | 分析 |
|------|-----|------|
| type | `range` | 索引范围扫描 |
| key | `idx_username` | 索引被正确使用 |
| key_len | `202` | username VARCHAR(50) + utf8mb4 (50*4+2=202) 参与索引范围扫描 |
| rows | `3` | 精确定位到 3 行 |
| Extra | `Using where` | 索引范围扫描后过滤 |

## 为什么快

`LIKE 'zhang%'` 的前缀是确定的（`zhang`），B+ 树可以定位到以 `zhang` 开头的位置，然后向后范围扫描直到遇到不以 `zhang` 开头的记录停止。

匹配到 3 行（zhang_san, zhang_si, zhang_wu），耗时 < 1ms。

> **注意：** `'%zhang%'` 匹配 5 行（包含中间出现的 zhang），`'zhang%'` 只匹配 3 行（仅开头）。如果业务确实需要全文搜索，应考虑全文索引（FULLTEXT INDEX）或 Elasticsearch。
