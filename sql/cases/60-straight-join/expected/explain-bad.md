# EXPLAIN 参考结果 - bad.sql（优化器选错驱动表，中间结果集爆炸）

## MySQL 8.0（实测 8.0.46，10 万订单 + 30 万订单项 + 1 万商品）

```
+----+-------------+-------+------------+--------+--------------------+----------------+---------+--------------------+--------+----------+-------------+
| id | select_type | table | partitions | type   | possible_keys      | key            | key_len | ref                | rows   | filtered | Extra       |
+----+-------------+-------+------------+--------+--------------------+----------------+---------+--------------------+--------+----------+-------------+
|  1 | SIMPLE      | i     | NULL       | ALL    | idx_order_id       | NULL           | NULL    | NULL               | 298734 |   100.00 | NULL        |
|  1 | SIMPLE      | p     | NULL       | eq_ref | PRIMARY,idx_category| PRIMARY       | 8       | test.i.product_id  |      1 |    10.00 | Using where |
|  1 | SIMPLE      | o     | NULL       | eq_ref | PRIMARY,idx_user_id| PRIMARY        | 8       | test.i.order_id    |      1 |     5.00 | Using where |
+----+-------------+-------+------------+--------+--------------------+----------------+---------+--------------------+--------+----------+-------------+
```

## 关键问题

| 字段 | 值 | 分析 |
|------|-----|------|
| 第一张表 `i` type | `ALL` | `t_order_item_sj`（30 万行）做驱动表，**全表扫描** |
| 第一张表 `i` rows | ~298,734 | 预估扫描约 30 万行，全部进入 JOIN 流水线 |
| 第二张表 `p` type | `eq_ref` | `p.id = i.product_id` 走主键等值查找，每行 1 次 |
| 第二张表 `p` filtered | `10.00` | `p.category = '电子'` 过滤，约 10% 命中 |
| 第三张表 `o` type | `eq_ref` | `o.id = i.order_id` 走主键等值查找，每行 1 次 |
| 第三张表 `o` filtered | `5.00` | `o.user_id = 100` 过滤，约 5% 命中 |
| JOIN 顺序 | item → product → order | 大表驱动，中间结果集巨大 |

## 为什么慢

优化器选择了 `t_order_item_sj → t_product_sj → t_order_sj` 的 JOIN 顺序：

1. 先全表扫描 `t_order_item_sj` 的约 30 万行，得到约 30 万行的中间结果集
2. 对这 30 万行的每一行，到 `t_product_sj` 用主键查 1 行（`eq_ref`），过滤 `category='电子'` 后剩约 3 万行
3. 再对这 3 万行的每一行，到 `t_order_sj` 用主键查 1 行（`eq_ref`），过滤 `user_id=100` 后只剩约 1500 行

问题在于：真正的过滤条件 `o.user_id = 100`（命中约 2 个订单）在**最后一步**才生效。前面两步 JOIN 都在围绕约 30 万行的中间结果集运作，绝大多数计算（30 万次 `t_product_sj` 主键查找 + 3 万次 `t_order_sj` 主键查找）最终被丢弃，属于纯浪费。

正确的思路应是"用小结果集驱动大表"：先把 `t_order_sj` 过滤到约 2 行，再用它驱动 `t_order_item_sj`、`t_product_sj`，每步结果集都很小（见 good.sql）。

实际耗时：约 **420 ms**。

## MySQL 5.7 差异

5.7 与 8.0 执行计划结构一致，优化器都可能选错驱动表。5.7 的统计信息不如 8.0 准确，选错驱动表的概率更高。
