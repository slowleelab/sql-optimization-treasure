# EXPLAIN 参考结果 - good.sql (前缀索引)

## MySQL 8.0（实测 8.0.46，15 万行数据）

```
+----+-------------+----------+------+-----------------+---------------+---------+-------+------+----------+-------+
| id | select_type | table    | type | possible_keys   | key           | key_len | ref   | rows | filtered | Extra |
+----+-------------+----------+------+-----------------+---------------+---------+-------+------+----------+-------+
|  1 | SIMPLE      | t_url_log| ref  | idx_url_prefix  | idx_url_prefix| 82      | const |    1 |   100.00 | NULL  |
+----+-------------+----------+------+-----------------+---------------+---------+-------+------+----------+-------+
```

## 关键改进

| 字段 | 值 | 分析 |
|------|-----|------|
| type | `ref` | 等值匹配前缀索引 |
| key | `idx_url_prefix` | 使用前缀索引 url(20) |
| key_len | **`82`** | **20×4+2 = 82 字节，缩减 92%** |
| rows | 1 | 命中 1 行（前缀选择性足够） |

## 为什么更好

前缀索引 `url(20)` 只索引 URL 的前 20 个字符：

1. **key_len 从 1022 降到 82**：索引体积缩小约 12 倍
2. **空间大幅节省**：15 万行索引空间从 ~150 MB 降到 ~12 MB
3. **buffer pool 友好**：小索引常驻内存，缓存命中率高
4. **写入更快**：B+ 树节点小，插入与分裂成本降低

## 前缀长度选择方法

```sql
-- 计算不同前缀长度的选择性，选择接近 1 的最小长度
SELECT
    COUNT(DISTINCT LEFT(url, 10)) / COUNT(*) AS sel_10,
    COUNT(DISTINCT LEFT(url, 20)) / COUNT(*) AS sel_20,
    COUNT(DISTINCT LEFT(url, 30)) / COUNT(*) AS sel_30,
    COUNT(DISTINCT url) / COUNT(*)          AS sel_full
FROM t_url_log;
```

| 前缀长度 | 选择性 | 评价 |
|----------|--------|------|
| 10 | ~0.50 | 区分度不足 |
| 20 | ~0.95 | 接近全列，推荐 |
| 30 | ~0.99 | 略有冗余 |
| full | 1.00 | 浪费空间 |

## 量化对比

| 指标 | bad.sql (全索引) | good.sql (前缀索引) | 提升 |
|------|------------------|---------------------|------|
| key_len | 1022 字节 | 82 字节 | **缩减 92%** |
| 索引空间 | ~150 MB | ~12 MB | **节省 ~92%** |
| 查询命中行数 | 1 | 1 | 无差异 |

::: tip 前缀索引适用场景
适合长字符串（URL、邮箱、路径）的等值/前缀匹配查询。注意：
- 不支持 `ORDER BY url`（索引不保证完整值有序）
- 不能作为覆盖索引
- 对 `WHERE url LIKE 'https://%'` 同样有效（前缀匹配）
:::
