# EXPLAIN 参考结果 - bad.sql（OR 条件中有一列无索引）

## MySQL 8.0（30 万行数据）

```
+----+-------------+----------+------------+------+---------------+------+---------+------+--------+----------+-------------+
| id | select_type | table    | partitions | type | possible_keys | key  | key_len | ref  | rows   | filtered | Extra       |
+----+-------------+----------+------------+------+---------------+------+---------+------+--------+----------+-------------+
|  1 | SIMPLE      | t_user_or| NULL       | ALL  | NULL          | NULL | NULL    | NULL | 299687 |    12.49 | Using where |
+----+-------------+----------+------------+------+---------------+------+---------+------+--------+----------+-------------+
```

## 关键问题

| 字段 | 值 | 分析 |
|------|-----|------|
| type | `ALL` | 全表扫描 |
| possible_keys | `NULL` | 虽然 phone 有 idx_phone，但 city 无索引，OR 导致整体无法走索引 |
| key | `NULL` | 无索引被使用 |
| rows | ~299,687 | 扫描全部 30 万行 |
| filtered | 12.49% | 绝大多数行被过滤掉 |
| Extra | `Using where` | 逐行检查 phone 或 city 条件 |

## 为什么慢

`WHERE phone = '13800138000' OR city = '北京'` 中，`phone` 有索引（idx_phone），但 `city` 没有。OR 条件要求两侧都能走索引才能使用 index_merge，否则优化器放弃索引走全表扫描。

30 万行逐行检查 OR 条件，耗时约 110ms。

## MySQL 5.7 差异

5.7 与 8.0 行为一致，均为全表扫描。
