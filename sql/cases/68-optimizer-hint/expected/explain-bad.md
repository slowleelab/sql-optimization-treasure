# EXPLAIN 参考结果 - bad.sql (优化器误选 idx_status，导致 filesort)

## MySQL 8.0（实测 8.0.46，100 万行订单数据）

```
-- EXPLAIN SELECT * FROM t_order WHERE user_id = 100 AND status = 1 ORDER BY created_at DESC LIMIT 10;
+----+-------------+---------+------------+------+-----------------------------------+-----------+---------+-------+--------+----------+----------------+
| id | select_type | table   | partitions | type | possible_keys                     | key       | key_len | ref   | rows   | filtered | Extra          |
+----+-------------+---------+------------+------+-----------------------------------+-----------+---------+-------+--------+----------+----------------+
|  1 | SIMPLE      | t_order | NULL       | ref  | idx_status,idx_user_created       | idx_status| 1       | const | 349872 |    10.00 | Using filesort |
+----+-------------+---------+------------+------+-----------------------------------+-----------+---------+-------+--------+----------+----------------+
```

### 对比：若优化器选对 idx_user_created（用 FORCE INDEX 模拟）

```
+----+-------------+---------+------------+------+--------------------------+------------------+---------+-------+------+----------+-------------+
| id | select_type | table   | partitions | type | possible_keys            | key              | key_len | ref   | rows | filtered | Extra       |
+----+-------------+---------+------------+------+--------------------------+------------------+---------+-------+------+----------+-------------+
|  1 | SIMPLE      | t_order | NULL       | ref  | idx_user_created         | idx_user_created | 8       | const |   10 |    10.00 | Using where |
+----+-------------+---------+------------+------+--------------------------+------------------+---------+-------+------+----------+-------------+
```

## 关键问题

| 字段 | 值（误选 idx_status） | 值（选对 idx_user_created） | 分析 |
|------|----------------------|----------------------------|------|
| type | `ref` | `ref` | 索引等值查找 |
| key | `idx_status` | `idx_user_created` | 错误 vs 正确的索引选择 |
| rows | ~349,872 | ~10 | status=1 扫描 35 万行 vs user_id=100 扫描 10 行 |
| filtered | 10.00 | 10.00 | 回表后过滤另一条件 |
| Extra | `Using filesort` | `Using where` | filesort vs 索引有序 |

## 为什么慢

### 误选 idx_status 的执行流程

1. 通过 `idx_status` 索引扫描 `status=1` 的约 35 万行索引记录
2. 每行回表查询完整行数据，过滤 `user_id = 100`
3. 过滤后仅剩约 10 行（user_id=100 且 status=1）
4. 对这 10 行按 `created_at DESC` 排序（filesort）
5. 取前 10 行返回

**问题**：扫描了 35 万行索引 + 35 万次回表，最终只需要 10 行。99.997% 的工作是浪费的。

### 选对 idx_user_created 的执行流程

1. 通过 `idx_user_created` 索引定位 `user_id=100` 的约 10 行索引记录
2. 索引本身按 `created_at` 有序，直接取最后 10 行（DESC 即反向读取）
3. 每行回表查询完整行数据，过滤 `status = 1`
4. 返回结果

**优势**：只扫描 10 行索引 + 10 次回表，索引有序无需 filesort。

### 优化器为什么会选错

优化器基于代价估算选择索引，估算依据是统计信息（ cardinality 、数据分布等）。以下情况可能导致误选：

1. **统计信息过期**：`ANALYZE TABLE` 未及时执行，cardinality 与实际不符
2. **数据分布变化**：status=1 的占比从 1% 增长到 35%，优化器仍按旧统计估算
3. **LIMIT 影响**：优化器知道 LIMIT 10，但低估了 idx_status 路径的回表代价
4. **代价模型缺陷**：MySQL 代价模型对回表代价的估算不够精确

实际耗时：
- 误选 idx_status：约 **850 ms**（35 万行扫描 + 回表 + filesort）
- 选对 idx_user_created：约 **0.5 ms**（10 行扫描 + 回表）

## 5.7 vs 8.0 差异

- 优化器选错索引的问题在两个版本中都存在
- 8.0 的代价模型略有改进，但统计信息不准确时仍会选错
- 8.0 支持 `EXPLAIN ANALYZE` 可查看实际执行代价，便于诊断
