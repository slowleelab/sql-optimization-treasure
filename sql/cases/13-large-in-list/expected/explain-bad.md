# EXPLAIN 参考结果 - bad.sql（大 IN 列表子查询查询目标用户订单）

## MySQL 8.0（20 万行订单数据）

```
+----+-------------+------------+------------+-------+---------------+-------------+---------+--------------------+--------+----------+------------------------------+
| id | select_type | table      | partitions | type  | possible_keys | key         | key_len | ref                |   rows | filtered | Extra                        |
+----+-------------+------------+------------+-------+---------------+-------------+---------+--------------------+--------+----------+------------------------------+
|  1 | PRIMARY     | t_order_in | NULL       | ALL   | idx_user      | NULL        | NULL    | NULL               | 200000 |   100.00 | Using where                  |
|  1 | PRIMARY     | <derived2> | NULL       | ref   | <auto_key0>   | <auto_key0> | 8       | t_order_in.user_id |      1 |   100.00 | NULL                         |
|  2 | DERIVED     | t_order_in | NULL       | index | idx_user      | idx_user    | 8       | NULL               | 200000 |   100.00 | Using index; Using temporary |
+----+-------------+------------+------------+-------+---------------+-------------+---------+--------------------+--------+----------+------------------------------+
```

## 关键问题

| 字段 | 值 | 分析 |
|------|-----|------|
| select_type | `PRIMARY` + `DERIVED` | 子查询被物化为派生表 `<derived2>` |
| type (t_order_in, PRIMARY) | `ALL` | 外层对订单表全表扫描 20 万行 |
| key (t_order_in, PRIMARY) | `NULL` | 外层查询未走索引，逐行用 WHERE 过滤 |
| rows (t_order_in, PRIMARY) | ~200,000 | 全表扫描订单表 |
| type (DERIVED) | `index` | 派生表通过 idx_user 覆盖索引扫描去重 |
| rows (DERIVED) | ~200,000 | 构建派生表也需扫描 20 万行索引 |
| Extra (DERIVED) | `Using index; Using temporary` | 覆盖索引去重，但需临时表物化 |

## 为什么慢

查询分两步执行。首先，派生子查询（`SELECT DISTINCT user_id FROM t_order_in LIMIT 1000`）被物化为派生表 `<derived2>`：需扫描 `idx_user` 覆盖索引的全部 20 万行条目，去重后取前 1000 个 user_id，写入临时表并自动建索引（`<auto_key0>`）。

然后，外层查询对 `t_order_in` 做**全表扫描**（`type=ALL`，20 万行），逐行用 `user_id IN (派生表)` 过滤。虽然派生表有自动索引可做 `ref` 等值探测，但外层仍是全表扫描驱动，每行都要回表读取完整数据（`SELECT *`）。

核心问题在于**驱动方向低效**：以 20 万行的大表为驱动表全扫描，而非以 1000 行的小集合为驱动表走索引查找。此外，物化派生表本身也有扫描和临时表开销。在实际业务中，若 IN 列表是上千个常量（而非子查询），还会有 SQL 解析膨胀和执行计划不稳定的问题。

实际耗时：约 **220 ms**（实测 MySQL 8.0，20 万行订单）。

## MySQL 5.7 差异

5.7 对派生表的处理类似，但 5.7 的派生表物化效率略低（缺少 8.0 的派生表合并优化），且大 IN 列表场景下优化器的 range 预估更容易失准，导致执行计划不稳定。8.0 在此场景已有一定改善，但根本的全表扫描驱动问题依然存在。
