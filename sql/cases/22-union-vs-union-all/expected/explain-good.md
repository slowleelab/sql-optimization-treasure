# EXPLAIN 参考结果 - good.sql (UNION ALL 不去重)

## MySQL 8.0（实测 8.0.46，各 10 万行数据）

```
+----+-------------+-------------+-------+---------------+----------+---------+------+--------+----------+-------------+
| id | select_type | table       | type  | possible_keys | key      | key_len | ref  | rows   | filtered | Extra       |
+----+-------------+-------------+-------+---------------+----------+---------+------+--------+----------+-------------+
|  1 | PRIMARY     | t_source_a  | index | NULL          | idx_code | 82      | NULL |  99812 |   100.00 | Using index |
|  2 | UNION       | t_source_b  | index | NULL          | idx_code | 82      | NULL |  99812 |   100.00 | Using index |
+----+-------------+-------------+-------+---------------+----------+---------+------+--------+----------+-------------+
```

## 关键改进

| 字段 | 值 | 分析 |
|------|-----|------|
| select_type | `PRIMARY` / `UNION` | **无 UNION RESULT 行！** |
| table A/B type | `index` | 全索引扫描（覆盖索引） |
| Extra | `Using index` | 覆盖索引，无需回表 |
| 临时表 | **无** | **`Using temporary` 消失** |

## 为什么更好

`UNION ALL` 不去重，直接拼接两个查询结果：

1. **无临时表**：不创建临时表，不写入去重
2. **无唯一键检查**：直接输出 A 表结果 + B 表结果
3. **流式输出**：A 表扫完即可输出，B 表扫完追加，无需等待全部完成

### 执行流程（优化后）

```
1. 扫描 t_source_a 的 idx_code 索引（10 万行）-> 直接输出
2. 扫描 t_source_b 的 idx_code 索引（10 万行）-> 直接输出
（无临时表、无去重）
```

## 量化对比

| 指标 | bad.sql (UNION) | good.sql (UNION ALL) | 提升 |
|------|-----------------|----------------------|------|
| 临时表 | Using temporary | 无 | **消除临时表** |
| 去重检查 | 20 万次 | 0 | **省去掉重开销** |
| 结果行数 | 20 万（无重复） | 20 万（无重复） | 一致 |
| 耗时 | ~120 ms | ~50 ms | **约 2.4 倍** |

## 何时该用 UNION（去重）

```sql
-- 场景: 两表可能有重复行，需要去重
SELECT code, name FROM t_source_a  -- 可能含 code='X01'
UNION
SELECT code, name FROM t_source_b  -- 也可能含 code='X01'
-- 此时 UNION 去重是必要的

-- 但更高效的做法：先确认是否有重复，能保证无重复就用 UNION ALL
-- 或者业务层去重，避免数据库做重活
```

## 决策表

| 场景 | 选择 | 原因 |
|------|------|------|
| 两结果集确定无重复 | **UNION ALL** | 避免去重开销 |
| 两结果集可能有重复 | UNION | 需要去重 |
| 需要去重但可接受业务层处理 | UNION ALL + 业务去重 | 数据库更轻量 |
| 单表查询用 UNION 拆分条件 | **UNION ALL** | 同表不可能自相重复 |

::: tip 经验法则
**默认用 UNION ALL**，只有明确需要去重且无法在数据层保证无重复时才用 UNION。
很多业务场景下，两表数据天然不交叉（如本案例 A/B 前缀），UNION 的去重纯属浪费。
:::
