# EXPLAIN 参考结果 - good.sql (优化后)

## MySQL 8.0（实测 8.0.46，约 49 万行数据）

```
+----+-------------+--------+-------+---------------+----------+---------+-------+------+----------+-------+
| id | select_type | table  | type  | possible_keys | key      | key_len | ref   | rows | filtered | Extra |
+----+-------------+--------+-------+---------------+----------+---------+-------+------+----------+-------+
|  1 | SIMPLE      | t_user | const | uk_phone      | uk_phone | 46      | const |    1 |   100.00 | NULL  |
+----+-------------+--------+-------+---------------+----------+---------+-------+------+----------+-------+
```

## 关键改进

| 字段 | 值 | 分析 |
|------|-----|------|
| type | `const` | **最优级别！** 唯一索引等值查找，最多匹配 1 行 |
| key | `uk_phone` | 走了唯一索引 |
| rows | **1** | 只扫描 1 行 |
| Extra | NULL | 无额外操作 |

## 为什么快

`WHERE phone = '13800138000'` 传入了字符串，类型与列定义 `VARCHAR(11)` 一致：

1. 无需类型转换 -> 索引正常工作
2. `uk_phone` 是唯一索引 -> `type=const`，直接定位到那一行
3. 只需 1 次索引查找，O(1) 复杂度

## 量化对比

| 指标 | bad.sql | good.sql | 提升 |
|------|---------|----------|------|
| type | ALL | const | 全表扫描 -> 常量查找 |
| rows | 495,079 | **1** | **495,079 倍** |
| key | NULL | uk_phone | 从不用索引到用唯一索引 |
| 耗时 | 90 ms | **1 ms** | **90 倍** |

::: tip 经验法则
永远确保查询条件的类型与列定义一致。VARCHAR 列就传字符串，INT 列就传数字。
这是最常见的索引失效原因之一，也是最容易修复的优化点。
:::
