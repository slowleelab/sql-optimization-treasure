# EXPLAIN 参考结果 - bad.sql (优化前)

## MySQL 8.0（实测 8.0.46，约 49 万行数据）

```
+----+-------------+--------+------+---------------+------+---------+------+--------+----------+-------------+
| id | select_type | table  | type | possible_keys | key  | key_len | ref  | rows   | filtered | Extra       |
+----+-------------+--------+------+---------------+------+---------+------+--------+----------+-------------+
|  1 | SIMPLE      | t_user | ALL  | uk_phone      | NULL | NULL    | NULL | 495079 |    10.00 | Using where |
+----+-------------+--------+------+---------------+------+---------+------+--------+----------+-------------+
```

## 关键问题

| 字段 | 值 | 分析 |
|------|-----|------|
| type | `ALL` | **全表扫描！** |
| possible_keys | `uk_phone` | 优化器知道有索引 |
| key | `NULL` | **但没用上！** |
| rows | ~495,079 | 扫描近 50 万行 |
| filtered | 10.00 | 仅 10% 的行可能匹配 |
| Extra | `Using where` | 需要逐行过滤 |

## 为什么慢

`phone` 是 `VARCHAR(11)`，查询 `WHERE phone = 13800138000` 传入了数字（没加引号）。

MySQL 的隐式类型转换规则：**字符串和数字比较时，把字符串转为数字**。

因此 MySQL 实际执行的是：
```sql
WHERE CAST(phone AS SIGNED) = 13800138000
```

对 `phone` 列套了 `CAST()` 函数，**索引失效**（函数操作破坏索引查找），退化为全表扫描，逐行 CAST 后比较。

实际耗时：约 **90 ms**（50 万行数据）。
