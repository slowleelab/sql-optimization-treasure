# 查询结果参考 - bad.sql（TIMESTAMP 在不同时区下读出值不同）

> 本案例重点不在 EXPLAIN 执行计划（两表都走 `idx_created` 范围扫描，执行计划无差异），
> 而在 **TIMESTAMP 受 session time_zone 影响导致同一行读出不同值**。以下展示 SELECT 结果对比。

## 场景说明

造数据时固定会话时区 `+00:00`，写入字面量 `'2026-07-01 08:00:00'`：

- `t_time_bad.created_at`（TIMESTAMP）：当作 UTC 08:00 写入，内部存 UTC 时间戳
- `t_time_good.created_at`（DATETIME）：原样存 `'2026-07-01 08:00:00'`

随后切换会话时区读取，观察 `created_at` 显示值是否随会话时区变化。

## 1. 同一行 created_at 在不同会话时区下的显示值（t_time_bad, TIMESTAMP）

```sql
SET SESSION time_zone = '+00:00';
SELECT 'UTC(+00:00)' AS session_tz, id, created_at FROM t_time_bad ORDER BY id LIMIT 3;

SET SESSION time_zone = '+08:00';
SELECT '+08:00' AS session_tz, id, created_at FROM t_time_bad ORDER BY id LIMIT 3;

SET SESSION time_zone = 'America/New_York';
SELECT 'America/New_York' AS session_tz, id, created_at FROM t_time_bad ORDER BY id LIMIT 3;
```

| session_tz | id | created_at | 与 UTC 偏差 |
|------------|----|------------|-------------|
| `+00:00` (UTC) | 1 | `2026-07-01 08:00:00` | 0 |
| `+08:00` (东八区) | 1 | `2026-07-01 16:00:00` | **+8 小时** |
| `America/New_York` (夏令时 UTC-4) | 1 | `2026-07-01 04:00:00` | **-4 小时** |

**关键现象**：同一行数据，内部存储不变，但会话时区一变，`created_at` 读出值就跟着变。东八区比 UTC 整整偏移 8 小时。

## 2. 报表 bug 复现："统计 2026-07-01 当天订单数"

数据分三批以 UTC 录入：

| 批次 | 行数 | UTC 时刻 | UTC 归属日 | +08:00 显示值 | +08:00 归属日 |
|------|------|----------|-----------|---------------|---------------|
| 批A | 800 | `2026-07-01 08:00:00` | 7-1 | `2026-07-01 16:00:00` | 7-1 |
| 批B | 100 | `2026-07-01 20:00:00` | 7-1 | `2026-07-02 04:00:00` | **7-2** |
| 批C | 100 | `2026-07-02 08:00:00` | 7-2 | `2026-07-02 16:00:00` | 7-2 |

```sql
-- UTC 会话统计 7-1 当天
SET SESSION time_zone = '+00:00';
SELECT COUNT(*) AS cnt_0701_utc
FROM t_time_bad
WHERE created_at >= '2026-07-01 00:00:00' AND created_at < '2026-07-02 00:00:00';

-- +08:00 会话统计 7-1 当天
SET SESSION time_zone = '+08:00';
SELECT COUNT(*) AS cnt_0701_plus8
FROM t_time_bad
WHERE created_at >= '2026-07-01 00:00:00' AND created_at < '2026-07-02 00:00:00';
```

| 会话时区 | 过滤区间（字面量） | cnt_0701 | 含义 |
|----------|--------------------|----------|------|
| `+00:00` | `2026-07-01 00:00 ~ 2026-07-02 00:00` | **900** | 批A 800 + 批B 100（正确归属） |
| `+08:00` | `2026-07-01 00:00 ~ 2026-07-02 00:00` | **800** | 仅批A，**批B 被推到 7-2，少了 100 行** |

**报表 bug 清晰可见**：同一批数据，UTC 会话统计 7-1 = 900，+08:00 会话统计 7-1 = 800，整整少了 100 行（批B）。批B 的 `2026-07-01 20:00:00` UTC 在 +08:00 会话下被读成 `2026-07-02 04:00:00`，归属日错误。

反向验证（统计 7-2 当天）：

| 会话时区 | cnt_0702 | 含义 |
|----------|----------|------|
| `+00:00` | **100** | 仅批C（正确归属） |
| `+08:00` | **200** | 批B 100 + 批C 100，**多算了批B** |

UTC 16:00~24:00 的订单在 +08:00 会话下被整体推到次日，**日报数据错位 8 小时**，晚间高峰订单归属错误最严重。

## 3. TIMESTAMP 的 2038 年问题（Y2038）

```sql
SELECT FROM_UNIXTIME(2147483647);  -- 2038-01-19 03:14:07 UTC，TIMESTAMP 上限
SELECT FROM_UNIXTIME(2147483648);  -- 越界：8.0 返回 NULL，5.7 置 0 并警告
```

| 时间点 | TIMESTAMP 表现 | DATETIME 表现 |
|--------|----------------|---------------|
| `2038-01-19 03:14:07 UTC` | 上限，可正常存储 | 可正常存储 |
| `2038-01-19 03:14:08 UTC` | **越界，写入失败/截断为 0** | 可正常存储 |
| `9999-12-31 23:59:59` | 不支持 | 上限，可正常存储 |

TIMESTAMP 内部用 4 字节有符号整数存"自 1970-01-01 UTC 起的秒数"，上限 `2^31-1 = 2147483647`。合同到期、长期会员、生日等远期/历史日期若用 TIMESTAMP，会在 2038 年后无法写入。

## 4. EXPLAIN 执行计划（无差异，仅供参考）

两表 `created_at` 均建了 `idx_created` 索引，范围扫描执行计划一致，本案例问题不在执行计划：

```
+----+-------------+------------+------------+-------+---------------+-------------+---------+------+------+----------+-----------------------+
| id | select_type | table      | partitions | type  | possible_keys | key         | key_len | ref  | rows | filtered | Extra                 |
+----+-------------+------------+------------+-------+---------------+-------------+---------+------+------+----------+-----------------------+
|  1 | SIMPLE      | t_time_bad | NULL       | range | idx_created   | idx_created | 4       | NULL |  900 |   100.00 | Using index condition |
+----+-------------+------------+------------+-------+---------------+-------------+---------+------+------+----------+-----------------------+
```

| 字段 | 值 | 分析 |
|------|-----|------|
| type | `range` | 索引范围扫描 |
| key | `idx_created` | 走 created_at 索引 |
| key_len | `4` | TIMESTAMP 占 4 字节 |
| rows | ~900 | 预估扫描 900 行 |
| Extra | `Using index condition` | 索引条件下推 |

## 为什么是"坏"方案

1. **同一行读出值随会话时区变化**：TIMESTAMP 读取时按 session time_zone 隐式转换，东八区比 UTC 偏移 8 小时，跨时区报表对不齐。
2. **报表切日错位 8 小时**：按 UTC 录入、按部署时区切日统计时，UTC 16:00~24:00 的订单归属日错误。
3. **2038 年上限**：4 字节存储导致远期日期无法写入，不适合长期业务表。
4. **行为隐式且不可控**：转换由会话时区配置隐式触发，SQL 文本相同但不同连接结果不同，难以测试和追溯。

## MySQL 5.7 差异

- 5.7 与 8.0 的 TIMESTAMP 时区转换机制一致（均按 session time_zone 双向转换）。
- 5.7 中 `time_zone='SYSTEM'` 时依赖操作系统时区；命名时区（如 `America/New_York`）需先用 `mysql_tzinfo_to_sql` 加载时区表。
- 5.7 对超出 TIMESTAMP 范围的值在严格 SQL 模式下报错，非严格模式下置 0 并警告；8.0 行为类似但警告信息更明确。
- 8.0 支持 `TIMESTAMP` 与 `DATETIME` 的自动初始化/更新（`DEFAULT CURRENT_TIMESTAMP` / `ON UPDATE`），5.7.2+ 起也支持，行为一致。
