# EXPLAIN 参考结果 - bad.sql (读走主库)

## MySQL 8.0（主库表 t_order_master，10 万行）

```
+----+-------------+----------------+------------+-------+-------------------+-------------------+---------+------+------+----------+-----------------------+
| id | select_type | table          | partitions | type  | possible_keys     | key               | key_len | ref  | rows | filtered | Extra                 |
+----+-------------+----------------+------------+-------+-------------------+-------------------+---------+------+------+----------+-----------------------+
|  1 | SIMPLE      | t_order_master | NULL       | ref   | idx_user_created  | idx_user_created  | 8       | const|    2 |   100.00 | Backward index scan   |
+----+-------------+----------------+------------+-------+-------------------+-------------------+---------+------+------+----------+-----------------------+
```

## 关键问题

| 字段 | 值 | 分析 |
|------|-----|------|
| type | `ref` | 走索引定位 user_id=12345 |
| key | `idx_user_created` | 用了 (user_id, created_at) 联合索引 |
| rows | ~2 | 预估命中约 2 行 |
| Extra | `Backward index scan` | 逆向扫描（ORDER BY created_at DESC） |

## 为什么慢

单看 EXPLAIN，这条查询本身并不慢——它走了索引、rows 很小。真正的代价在**架构层面**：

1. **读写争抢主库资源**：主库同时处理大量 INSERT/UPDATE 和 SELECT，CPU、内存、I/O 被读写两类负载共享，互相挤占
2. **锁竞争**：写操作持有的行锁/间隙锁会阻塞读查询（或反过来），高并发下等待时间显著增加
3. **连接数瓶颈**：主库 max_connections 有限，大量读连接占用连接数，导致写入请求排队甚至被拒绝
4. **Buffer Pool 污染**：大量扫描型读查询把冷数据页加载进 Buffer Pool，把热点写数据挤出缓存，写入变慢
5. **无法横向扩展**：主库单点，读 QPS 上限就是单机上限，无法通过加机器提升读吞吐

### 生产环境的真实问题

```
未分离（单主库承载全部流量）:
  写入峰值 5000 QPS + 读取峰值 50000 QPS
    -> 主库 CPU 长期 >80%，写入响应时间抖动
    -> 大量读连接占用连接池，写入排队
    -> Buffer Pool 被读查询污染，写入命中率下降
    -> 单机读上限就是系统读上限，无法扩展

分离后（1主 + 3从）:
  主库: 仅承载写入 + 强一致读（约 5000 QPS）
  从库: 承载读流量（3 台分担，每台约 17000 QPS）
    -> 主库写入稳定，无读干扰
    -> 读吞吐随从库数量线性扩展
    -> 故障隔离: 从库慢查询不影响主库写入
```

实际耗时：约 **4 ms**（实测 MySQL 8.0.46，10 万行表）。
在读写混合高并发场景下，主库因争抢资源，同类查询可能抖动到 **20-50 ms**。

## MySQL 5.7 差异

5.7 行为一致。Extra 显示 `Using filesort` 而非 `Backward index scan`（5.7 无降序索引优化）。
读写分离的架构价值与版本无关。
