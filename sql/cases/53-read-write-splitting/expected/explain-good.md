## 为什么快

执行计划结构与 bad 完全一致（两张表同构、数据相同），性能差异完全来自**架构层面的负载分离**：

1. **读写互不干扰**：从库只处理读，没有写入争抢锁、CPU、I/O，读查询响应稳定无抖动
2. **Buffer Pool 专属**：从库 Buffer Pool 全部用于缓存读数据，命中率更高
3. **连接数独立**：从库有独立的连接池，读连接再多也不影响主库写入
4. **横向扩展**：读吞吐不够时加从库即可线性提升，主库写入能力不受影响
5. **故障隔离**：从库上的慢查询、大扫描不会拖垮主库的写入链路

### 读写分离的核心价值

```
分离前: 单主库
  读写混合 -> 互相干扰 -> 写入抖动 + 读延迟不稳定
  读上限 = 单机上限，无法扩展

分离后: 1主 N从
  主库: 专注写入，响应稳定可预测
  从库: 专注读取，按需横向扩展
  -> 读吞吐随从库数量线性增长
  -> 写入性能不受读流量影响
```

### 复制延迟问题（重要）

读写分离引入的**核心风险是主从复制延迟**：

```
用户下单（写主库）-> 主库 binlog -> 异步复制 -> 从库 relay log -> 应用 -> 从库数据更新
                           ^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^
                           这段时间内，读从库会读到旧数据（不一致）

延迟来源:
  1. 网络传输 binlog 的延迟
  2. 从库回放 relay log 的速度（大事务会严重拖慢）
  3. 从库自身查询负载过高，SQL 线程拿不到 CPU

典型延迟: 毫秒级（正常） ~ 秒级（大事务/高负载）
```

### 解决复制延迟的常见策略

1. **关键路径读主库**：下单后立即查询、支付后查询余额等强一致场景，强制读主库
2. **写后短时读主**：写入后 N 秒内（如 1-3 秒）相关查询走主库，过了窗口再读从库
3. **半同步复制**：主库写入后至少等待一个从库确认收到 binlog 才返回，降低数据丢失风险（但不消除延迟）
4. **从库延迟监控**：监控 `Seconds_Behind_Master`，延迟超阈值时读请求自动降级到主库
5. **避免大事务**：大事务导致 binlog 膨胀、从库回放慢，应拆分批量操作

## 量化对比

| 指标 | bad (读主库) | good (读从库) | 提升 |
|------|--------------|---------------|------|
| 写读隔离 | 读写混合争抢资源 | 完全隔离 | 写入稳定 |
| 读吞吐上限 | 单机 | 随从库线性扩展 | N 倍 |
| 主库连接占用 | 高 | 低（仅写入+强一致读） | 释放连接 |
| 一致性 | 强（读主库） | 最终一致（有延迟） | 需权衡 |
| 单次查询耗时 | ~4 ms（可能抖动到 50ms） | ~3 ms（稳定） | 更稳定 |

> 本案例单表数据量小，单次查询差异不明显。读写分离的价值在高并发、大数据量场景下才充分体现：主库写入稳定 + 读吞吐可扩展。

## 5.7 vs 8.0 差异

- 执行计划结构一致，读写分离方案在两个版本上都有效
- 8.0 支持并行复制（MTS, Multi-Threaded Slave），从库回放 binlog 更快，复制延迟更低
- 8.0 支持 WRITESET 并行复制，进一步降低延迟
- 核心价值在于架构层面的读写分离，与版本无关

## 避坑指南

1. **不要盲目读从库**：强一致场景（下单后查询、支付确认）必须读主库，否则会因复制延迟读到旧数据
2. **监控复制延迟**：`SHOW SLAVE STATUS` 中的 `Seconds_Behind_Master` 要纳入监控告警
3. **从库也要建索引**：从库索引缺失会导致读查询变慢，DDL 要同步到所有从库
4. **负载均衡要均匀**：多从库时用代理（如 ProxySQL/HAProxy）做读流量分发，避免单台从库过载
5. **大事务会拖慢复制**：批量 DELETE/UPDATE 拆小批，避免单个大事务阻塞从库回放
6. **主从切换要演练**：主库故障时提升从库为新主库，需提前演练切换流程和数据校验
7. **考虑读写分离中间件**：ShardingSphere-Proxy、ProxySQL、MyCat 等可自动路由读写，降低应用层复杂度
