## 关键问题

| 字段 | 值 | 分析 |
|------|-----|------|
| type | `ref` | 走了索引定位 category |
| key | `idx_category_created` | 索引正常 |
| rows | ~20,000 | 预估扫描行数正常 |
| 表大小 | 512 MB | 10 万行占用 512 MB，平均每行 5KB |

## 为什么慢

虽然列表查询没有 `SELECT content`，但 TEXT 字段的存在仍拖慢查询：

1. **每页行数少**：InnoDB 默认页 16KB，bad 表每行约 5KB（含 TEXT 指针和部分行内数据），每页只能放约 3 行
2. **扫描更多数据页**：取 20 行需要读取约 7 个数据页，而 good 表只需 1 页
3. **Buffer Pool 浪费**：512 MB 的表占用大量 Buffer Pool 空间，冷数据（TEXT）挤走热数据（元数据页）
4. **回表代价高**：通过二级索引回表到聚簇索引时，即使不读 TEXT 字段，InnoDB 仍需加载包含 TEXT 指针的完整数据页

实际耗时：约 **45 ms**（列表查询，不查 TEXT）。

## MySQL 5.7 差异

5.7 行格式默认 DYNAMIC，TEXT 存储机制与 8.0 一致（行内存指针，内容存溢出页）。性能表现类似。
