## 关键改进

| 指标 | bad (混在一起) | good (拆表后) | 分析 |
|------|----------------|---------------|------|
| 主表大小 | 512 MB | **28.5 MB** | 减少 94%（TEXT 拆走了） |
| 每页行数 | ~3 行 | **~80 行** | 16KB 页 / 0.2KB 行 |
| 列表查询扫描页数 | ~7 页 | **~1 页** | 减少 85% |
| Buffer Pool 效率 | 低（TEXT 挤占） | **高**（只有热数据） |
| 列表查询耗时 | ~45 ms | **~8 ms** | 5.6 倍提升 |

## 为什么快

拆表后的核心收益不是"少读几个字段"，而是**整个数据页结构的变化**：

1. **主表更紧凑**：去掉 5KB 的 TEXT 后，每行约 0.2KB，16KB 页可放约 80 行（bad 表只能放 3 行）
2. **列表查询只读主表**：主表 28.5 MB 完全可以常驻 Buffer Pool，命中率接近 100%
3. **Buffer Pool 不被冷数据污染**：正文 TEXT（485 MB）只在详情页读取，不占用列表查询的 Buffer Pool 空间
4. **回表代价降低**：回表读取的聚簇索引页更紧凑，I/O 量更小
5. **详情页精确取一条**：通过主键 JOIN 扩展表，`type=eq_ref` 只读 1 行 TEXT，代价可控

## 量化对比

| 指标 | bad (混合表) | good (拆表后) | 提升 |
|------|-------------|---------------|------|
| 主表大小 | 512 MB | 28.5 MB | **减少 94%** |
| 每页行数 | ~3 | ~80 | **27 倍** |
| 列表查询耗时 | 45 ms | 8 ms | **5.6 倍** |
| Buffer Pool 命中率 | ~70% | ~99% | **+29%** |
| 详情页耗时 | 5 ms | 6 ms | 持平（多一次 JOIN） |

## 5.7 vs 8.0 差异

- 垂直拆表方案与 MySQL 版本无关，5.7 和 8.0 均可使用
- 8.0 的 Buffer Pool 管理更智能，拆表后命中率提升更明显
- 5.7 和 8.0 默认行格式均为 DYNAMIC，TEXT 存储机制一致
