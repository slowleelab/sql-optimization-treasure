## 为什么快

### 8.0 派生条件下推的执行流程

1. 优化器将外层 `WHERE t.user_id = 100` 下推到派生表内部
2. 派生表等价于：`SELECT user_id, SUM(amount) FROM t_order WHERE user_id = 100 GROUP BY user_id`
3. 通过 idx_user_id 索引精确查找 user_id=100 的约 10 行订单
4. 对这 10 行分组聚合，只产出 1 行结果
5. 外层直接返回，无需过滤

**核心优势**：
- 只扫描 10 行索引（vs 5.7 的 100 万行）
- 只物化 1 行结果（vs 5.7 的 10 万行）
- 无需外层过滤（vs 5.7 的 90% 过滤）

### 派生条件下推的适用条件

8.0 的派生条件下推并非万能，以下条件需满足：

1. **简单条件**：等值（=）、范围（>、<、BETWEEN）等简单条件可下推
2. **无聚合函数混合**：派生表 SELECT 列表包含聚合函数时，下推可能受限
3. **无 HAVING**：派生表包含 HAVING 时，条件下推可能失败
4. **无 LIMIT**：派生表包含 LIMIT 时，条件下推可能改变语义

```sql
-- 可下推：简单等值条件
SELECT * FROM (SELECT user_id, SUM(amount) FROM t_order GROUP BY user_id) t WHERE t.user_id = 100;

-- 可能不可下推：包含 HAVING
SELECT * FROM (SELECT user_id, SUM(amount) FROM t_order GROUP BY user_id HAVING SUM(amount) > 1000) t WHERE t.user_id = 100;

-- 可能不可下推：包含 LIMIT
SELECT * FROM (SELECT user_id, SUM(amount) FROM t_order GROUP BY user_id LIMIT 1000) t WHERE t.user_id = 100;
```

### 5.7 中的手动优化

5.7 不支持自动下推，需手动将条件下推到子查询内部：

```sql
-- 5.7 手动优化：将条件下推到子查询内部
SELECT user_id, SUM(amount) AS total
FROM t_order
WHERE user_id = 100
GROUP BY user_id;
```

## 5.7 vs 8.0 差异

- **5.7**：派生表总是全量物化，外层 WHERE 无法下推，需手动改写 SQL
- **8.0**：优化器自动下推简单条件，无需修改 SQL，性能提升显著
- **8.0 局限**：复杂派生表（含 HAVING、LIMIT、聚合函数混合）可能无法下推

::: tip 派生表优化原则
1. **8.0 用户**：优先依赖优化器的自动下推，但需验证 EXPLAIN 确认下推生效
2. **5.7 用户**：手动将条件下推到子查询内部，避免全量物化
3. **通用原则**：尽量避免在派生表外层过滤，改为在子查询内部过滤
4. **复杂场景**：当派生表包含 HAVING、LIMIT 时，考虑改写为 JOIN 或 CTE
:::
