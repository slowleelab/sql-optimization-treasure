# EXPLAIN 参考结果 - bad.sql（范围查询后列索引失效）

## MySQL 8.0（30 万行数据）

```
+----+-------------+---------------+------------+------+------------------------+------------------------+---------+-------+------+----------+-----------------------+
| id | select_type | table         | partitions | type | possible_keys          | key                    | key_len | ref   | rows | filtered | Extra                 |
+----+-------------+---------------+------------+------+------------------------+------------------------+---------+-------+------+----------+-----------------------+
|  1 | SIMPLE      | t_order_range | NULL       | ref  | idx_user_status_amount | idx_user_status_amount | 9       | const |    3 |    33.33 | Using index condition |
+----+-------------+---------------+------------+------+------------------------+------------------------+---------+-------+------+----------+-----------------------+
```

## 关键问题

| 字段 | 值 | 分析 |
|------|-----|------|
| type | `ref` | 走了索引（通过 user_id 等值定位） |
| key | `idx_user_status_amount` | 使用了联合索引 |
| key_len | `9` | user_id(8) + status(1) = 9 字节，**amount 列未参与索引** |
| rows | `3` | user_id=1000 且 status>1 匹配 3 行 |
| filtered | 33.33% | 还需在回表后过滤 amount>500 的条件 |

## 为什么慢

联合索引 `(user_id, status, amount)` 中，`status > 1` 是范围查询。MySQL 联合索引中，**范围查询之后的列无法参与索引过滤**。

所以 `user_id=1000` 走等值、`status>1` 走范围，但 `amount>500` 只能在回表后用 WHERE 逐行检查，无法利用索引。key_len = 9（user_id BIGINT 8 字节 + status TINYINT 1 字节），说明只有前两列被用到了。

虽然此案例因 user_id=1000 的数据量少（3 行）所以绝对耗时不高，但在生产环境中 user_id 下有大量数据时，范围查询后列失效会导致大量无效回表。

## MySQL 5.7 差异

5.7 与 8.0 行为一致，key_len 同样为 9，amount 列不参与索引。
