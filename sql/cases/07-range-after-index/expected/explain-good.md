# EXPLAIN 参考结果 - good.sql（status 改为等值条件，三列全走索引）

## MySQL 8.0（30 万行数据）

```
+----+-------------+--------------+------------+------+-------------------------+-------------------------+---------+-------------------+------+----------+-----------------------+
| id | select_type | table        | partitions | type | possible_keys           | key                     | key_len | ref               | rows | filtered | Extra                 |
+----+-------------+--------------+------------+------+-------------------------+-------------------------+---------+-------------------+------+----------+-----------------------+
|  1 | SIMPLE      | t_order_range| NULL       | ref  | idx_user_status_amount  | idx_user_status_amount  | 14      | const,const,const |    2 |    66.67 | Using index condition |
+----+-------------+--------------+------------+------+-------------------------+-------------------------+---------+-------------------+------+----------+-----------------------+
```

## 关键改进

| 字段 | 值 | 分析 |
|------|-----|------|
| type | `ref` | 索引等值查找 |
| key | `idx_user_status_amount` | 使用了联合索引 |
| key_len | `14` | user_id(8) + status(1) + amount(5) = **14 字节，三列全部参与索引** |
| rows | `2` | 匹配 2 行（status=2 且 amount>500） |
| filtered | 66.67% | 仍有 amount 范围过滤，但大部分在索引层完成 |

## 为什么快

将 `status > 1`（范围查询）改为 `status = 2`（等值查询）后，`status` 不再是范围条件，其后的 `amount > 500` 可以继续参与索引过滤。

key_len 从 9 增加到 14，说明 `amount` 列也加入了索引扫描，减少了回表次数。

::: tip 生产建议
如果业务确实需要对 status 做范围查询，应调整联合索引列顺序，把范围列放最后：`(user_id, amount, status)`。
:::

## 量化对比

| 指标 | bad.sql | good.sql | 提升 |
|------|---------|----------|------|
| key_len | 9 | 14 | 多用了一列索引 |
| rows | 4 | 2 | 减少 50% |
| filtered | 33.33% | 66.67% | 过滤效率翻倍 |
