## 关键改进

| 字段 | bad (全分片扫描) | good (精确路由) | 分析 |
|------|-----------------|----------------|------|
| 查询分片数 | 4 个 | **1 个** | 只访问目标分片 |
| 网络 RTT | 4 次 | **1 次** | 减少 75% 网络开销 |
| 连接占用 | 4 个分片连接 | **1 个分片连接** | 释放 75% 连接 |
| 无效扫描 | 3 个分片 | **0 个** | 消除浪费 |
| 结果合并 | UNION ALL 合并 | **无需合并** | 减少 CPU 开销 |

## 为什么快

路由计算将"广播查询"变为"精确查询"：

1. **应用层路由**：`shard = user_id % 4`，user_id=100 -> shard=0 -> 直接查 t_order_0
2. **单次查询**：只访问 1 个分片，1 次网络 RTT，1 个连接
3. **与分片数无关**：无论 4 个分片还是 64 个分片，单次查询只访问 1 个分片
4. **性能等价单表**：查询性能与单表查询完全一致，分片带来的水平扩展优势得以保留

### 路由策略对比

```
无路由（广播）:
  查询 -> 所有分片 -> 合并结果
  耗时 = N 个分片 × 单次查询耗时
  分片越多越慢

有路由（精确）:
  查询 -> 路由计算 -> 目标分片 -> 返回结果
  耗时 = 1 次单次查询耗时
  分片数不影响单次查询性能
```

### 常见路由策略

| 策略 | 路由方式 | 优点 | 缺点 |
|------|----------|------|------|
| 取模 | `user_id % N` | 简单均匀 | 扩容需迁移大量数据 |
| 一致性哈希 | `hash(user_id) % 虚拟节点` | 扩容迁移少 | 实现复杂 |
| 范围分片 | `user_id / 1000000` | 范围查询友好 | 容易数据倾斜 |
| 基因法 | user_id 嵌入订单号 | 订单号也能路由 | 需要 ID 生成器配合 |

## 5.7 vs 8.0 差异

- 执行计划结构一致，路由策略与 MySQL 版本无关
- 核心价值在于应用层的分片路由设计
- 8.0 的并行查询能力对 UNION ALL 略有优化，但不改变架构层面的问题

## 避坑指南

1. **路由键选择**：选择查询频率最高的字段作为路由键（如 user_id），避免跨分片查询
2. **路由键嵌入 ID**：将路由基因嵌入订单号（如订单号包含 user_id % 4），避免按订单号查询时广播
3. **避免跨分片 JOIN**：不同分片的数据无法直接 JOIN，需要应用层组装
4. **扩容要提前规划**：取模路由扩容需迁移数据，一致性哈希或范围分片扩容更友好
5. **使用分片中间件**：ShardingSphere-JDBC、MyCat 等可自动处理路由，降低应用层复杂度
6. **监控分片倾斜**：定期检查各分片数据量，避免某个分片过大成为瓶颈
