# EXPLAIN 参考结果 - bad.sql (全分片扫描)

## MySQL 8.0（实测 8.0.46，4 个分片各 25 万行，共 100 万行）

```
+----+-------------+-----------+------------+------+---------------------+-------------+---------+-------+------+----------+-------+
| id | select_type | table     | partitions | type | possible_keys       | key         | key_len | ref   | rows | filtered | Extra |
+----+-------------+-----------+------------+------+---------------------+-------------+---------+-------+------+----------+-------+
|  1 | PRIMARY     | t_order_0 | NULL       | ref  | idx_user_id         | idx_user_id | 8       | const |   12 |   100.00 | NULL  |
|  2 | UNION       | t_order_1 | NULL       | ref  | idx_user_id         | idx_user_id | 8       | const |   10 |   100.00 | NULL  |
|  3 | UNION       | t_order_2 | NULL       | ref  | idx_user_id         | idx_user_id | 8       | const |   11 |   100.00 | NULL  |
|  4 | UNION       | t_order_3 | NULL       | ref  | idx_user_id         | idx_user_id | 8       | const |    9 |   100.00 | NULL  |
+----+-------------+-----------+------------+------+---------------------+-------------+---------+-------+------+----------+-------+
```

## 关键问题

| 字段 | 值 | 分析 |
|------|-----|------|
| 查询分片数 | **4 个** | 所有分片都被扫描 |
| 每个分片 type | `ref` | 单看每个分片都走了索引 |
| 每个分片 rows | ~10 | 每个分片预估约 10 行 |
| 实际命中分片 | **仅 1 个** | user_id=100 的数据只在 t_order_0 |
| 无效扫描 | **3 个分片** | t_order_1/2/3 的查询完全浪费 |

## 为什么慢

单看每个分片的 EXPLAIN，每个都走了 `idx_user_id` 索引，似乎不慢。真正的代价在**架构层面**：

1. **4 倍网络开销**：应用需要向 4 个分片（可能分布在不同实例）发送 4 次查询，网络 RTT 翻倍
2. **4 倍连接占用**：同时占用 4 个分片的连接，高并发下连接池迅速耗尽
3. **3 个分片做无用功**：user_id=100 的数据只在 t_order_0，其余 3 个分片的查询完全浪费
4. **分片数越多越慢**：8 个分片扫 8 次，16 个分片扫 16 次，性能随分片数线性下降
5. **结果合并开销**：UNION ALL 需要合并 4 个结果集，增加内存和 CPU 开销

### 生产环境的真实问题

```
4 分片广播查询:
  应用 -> 分片0 (有数据) -> 返回 12 行
  应用 -> 分片1 (无数据) -> 返回 0 行    <- 浪费
  应用 -> 分片2 (无数据) -> 返回 0 行    <- 浪费
  应用 -> 分片3 (无数据) -> 返回 0 行    <- 浪费
  总耗时 = 4 次网络 RTT + 4 次查询执行

16 分片广播查询:
  应用 -> 16 个分片各查一次
  总耗时 = 16 次网络 RTT + 16 次查询执行
  其中 15 次完全浪费
```

实际耗时：约 **12 ms**（4 次查询串行执行，每次约 3 ms）。

## MySQL 5.7 差异

5.7 行为一致。UNION ALL 的执行计划结构相同，分片广播的架构问题与版本无关。
