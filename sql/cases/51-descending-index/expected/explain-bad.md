# EXPLAIN 参考结果 - bad.sql（升序索引，ORDER BY DESC）

## MySQL 5.7（DESC 索引被忽略，按 ASC 存储）

5.7 中 `idx_type_created (event_type, created_at)` 的 `created_at` 列实际按 ASC 升序存储（5.7 会忽略建索引时写的 DESC 关键字），且 5.7 不支持反向索引扫描优化。

```
+----+-------------+--------------+------+------------------+------------------+---------+-------+--------+----------+----------------+
| id | select_type | table        | type | possible_keys     | key              | key_len | ref   | rows   | filtered | Extra          |
+----+-------------+--------------+------+------------------+------------------+---------+-------+--------+----------+----------------+
|  1 | SIMPLE      | t_event_log  | ref  | idx_type_created | idx_type_created | 82      | const |  91800 |   100.00 | Using filesort |
+----+-------------+--------------+------+------------------+------------------+---------+-------+--------+----------+----------------+
```

## MySQL 8.0（实测 8.0.46，20 万行数据，仅有升序索引）

8.0 支持反向索引扫描（Backward index scan），即使索引是 ASC 也能逆向遍历，不需要 filesort。但仍非最优方案。

```
+----+-------------+--------------+------+------------------+------------------+---------+-------+--------+----------+-----------------------+
| id | select_type | table        | type | possible_keys     | key              | key_len | ref   | rows   | filtered | Extra                 |
+----+-------------+--------------+------+------------------+------------------+---------+-------+--------+----------+-----------------------+
|  1 | SIMPLE      | t_event_log  | ref  | idx_type_created | idx_type_created | 82      | const |  91800 |   100.00 | Backward index scan   |
+----+-------------+--------------+------+------------------+------------------+---------+-------+--------+----------+-----------------------+
```

## 关键问题

| 字段 | 值 | 分析 |
|------|-----|------|
| type | `ref` | 通过索引等值查找 event_type |
| key | `idx_type_created` | 用了升序联合索引 |
| rows | ~91,800 | event_type='LOGIN' 匹配约 9 万行 |
| Extra (5.7) | **`Using filesort`** | **5.7 需要额外排序！** |
| Extra (8.0) | `Backward index scan` | 8.0 反向扫描索引（无需 filesort，但仍非最优） |

## 为什么慢

**5.7 的问题**：`idx_type_created (event_type, created_at)` 的 `created_at` 列按 ASC 升序存储。查询 `ORDER BY created_at DESC` 需要倒序输出，但 5.7 无法高效地反向遍历索引 + LIMIT 提前终止，只能扫描匹配行后做 filesort 排序。

**8.0 的改进**：8.0 支持 `Backward index scan`（反向索引扫描），可以逆向遍历升序索引来满足 `ORDER BY DESC`，消除了 filesort。但反向扫描相比正向扫描仍有额外开销，且无法像降序索引那样原生有序。

实际耗时（8.0，仅升序索引）：约 **4 ms**。

::: warning 5.7 的陷阱
5.7 允许在建索引时写 `created_at DESC`，但不报错且不生效，索引仍按 ASC 存储。这是最常见的"降序索引失效"误区。8.0 才真正支持降序索引的物理存储。
:::
