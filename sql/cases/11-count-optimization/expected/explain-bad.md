# EXPLAIN 参考结果 - bad.sql（大表 COUNT(*) 统计已支付订单数）

## MySQL 8.0（50 万行订单数据）

```
+----+-------------+---------------+------------+------+---------------+------------+---------+-------+--------+----------+-------------+
| id | select_type | table         | partitions | type | possible_keys | key        | key_len | ref   |   rows | filtered | Extra       |
+----+-------------+---------------+------------+------+---------------+------------+---------+-------+--------+----------+-------------+
|  1 | SIMPLE      | t_order_count | NULL       | ref  | idx_status    | idx_status | 1       | const | 125000 |   100.00 | Using index |
+----+-------------+---------------+------------+------+---------------+------------+---------+-------+--------+----------+-------------+
```

## 关键问题

| 字段 | 值 | 分析 |
|------|-----|------|
| type | `ref` | 通过 idx_status 索引定位 status=1 |
| key | `idx_status` | 使用了 status 索引 |
| rows | ~125,000 | 预估扫描约 12.5 万行索引条目（50 万 / 4 个状态） |
| Extra | `Using index` | 覆盖索引，无需回表 |

## 为什么慢

虽然走了 `idx_status` 覆盖索引（`Using index`，无需回表），但 `COUNT(*)` 仍需**逐条扫描索引中所有 `status=1` 的条目**来计数。50 万行数据中约 1/4（约 12.5 万行）满足 `status=1`，MySQL 必须遍历这 12.5 万个索引条目才能得到结果。

核心问题在于**复杂度是 O(n)**：随着订单数据线性增长，扫描行数同步增长，统计接口越来越慢。即使覆盖索引避免了回表 I/O，索引扫描本身的 CPU 和缓冲池开销仍然随数据量线性增加。

这是典型的"统计查询反模式"：实时 COUNT 大表，将本应预计算的工作推迟到查询时完成。

实际耗时：约 **85 ms**（实测 MySQL 8.0，50 万行订单）。

## MySQL 5.7 差异

5.7 执行计划与 8.0 一致（`type=ref`，`Using index`），性能表现基本相同。问题根源是算法层面的 O(n) 扫描，与版本无关。
