# EXPLAIN 参考结果 - good.sql (RC 隔离级别，仅记录锁，不阻塞插入)

## MySQL 8.0（50 万行订单数据，RC 隔离级别）

执行 setup-good.sql 后，当前会话隔离级别已切换为 READ COMMITTED。

```
-- EXPLAIN SELECT * FROM t_order WHERE user_id = 100 AND status = 1 FOR UPDATE;
+----+-------------+---------+------------+------+--------------------------+------------------+---------+-------------+------+----------+-------------+
| id | select_type | table   | partitions | type | possible_keys            | key              | key_len | ref         | rows | filtered | Extra       |
+----+-------------+---------+------------+------+--------------------------+------------------+---------+-------------+------+----------+-------------+
|  1 | SIMPLE      | t_order | NULL       | ref  | idx_user_status          | idx_user_status  | 9       | const,const |    4 |   100.00 | Using where |
+----+-------------+---------+------------+------+--------------------------+------------------+---------+-------------+------+----------+-------------+
```

执行计划与 bad.sql 相同（都是索引等值查找），**优化点不在于执行计划，而在于隔离级别降低后锁范围缩小**。

## 关键改进

| 维度 | bad.sql（RR） | good.sql（RC） |
|------|--------------|---------------|
| 隔离级别 | REPEATABLE-READ | READ-COMMITTED |
| 锁类型 | next-key lock（记录锁+间隙锁） | 仅记录锁 |
| 锁范围 | 索引区间（含间隙） | 仅命中的行 |
| 并发插入 | 被阻塞 | 不受影响 |
| 幻读 | 防止 | 可能出现 |

## 为什么快

### RC 下的锁范围（仅记录锁）

在 RC 隔离级别下，同样的查询只加记录锁：

```
索引 idx_user_status 上的记录（user_id=100 的部分）：

  (100, 0)  (100, 0)  (100, 1)  (100, 1)  (100, 1)  (100, 1)  (100, 2)  (100, 3)
     |         |         |         |         |         |         |         |
   记录      记录      记录      记录      记录      记录      记录      记录

锁范围：
  - 记录锁：仅 (100,1) 的 4 条记录
  - 间隙锁：无

  实际锁定：仅 4 条匹配的记录
```

### 插入不被阻塞时间线

```
时间线   会话A（RC，加锁）                        会话B（不受影响）
  T1     SET SESSION TRANSACTION ISOLATION
         LEVEL READ COMMITTED;
  T2     BEGIN;
  T3     SELECT ... WHERE user_id=100 AND status=1
         FOR UPDATE;
         -- 仅持有 (100,1) 的 4 条记录锁
  T4                                          BEGIN;
  T5                                          INSERT INTO t_order
                                              (order_no, user_id, amount, status)
                                              VALUES ('NO999999', 100, 99.00, 0);
                                              -- 新记录 (100, 0) 不在锁定范围内
                                              -- 插入成功！
  T6                                          COMMIT;
  T7     COMMIT;
         => 会话B 正常执行，不受会话A 影响
```

### RC vs RR 锁行为对比

| 场景 | RR（可重复读） | RC（读已提交） |
|------|---------------|---------------|
| 等值查询命中唯一索引 | 记录锁 | 记录锁 |
| 等值查询命中普通索引 | next-key lock | 记录锁 |
| 范围查询 | next-key lock | 记录锁 |
| 间隙锁 | 有 | 无 |
| 幻读 | 防止 | 可能出现 |

### RC 的适用场景

1. **高并发写入**：RC 不加间隙锁，并发插入不受影响，适合写入密集型业务
2. **不需要防止幻读**：业务能容忍幻读（同一事务内两次查询结果可能不同）
3. **互联网应用**：大多数互联网应用使用 RC，配合乐观锁/悲观锁保证一致性

### RC 的注意事项

1. **幻读**：同一事务内两次执行相同查询，可能看到不同结果（其他事务插入了新数据）
2. **binlog 格式**：RC 下需使用 ROW 格式 binlog，否则可能导致主从数据不一致
3. **一致性要求高的场景**：如金融交易，仍需使用 RR 或 Serializable

```sql
-- 查看 binlog 格式（RC 下必须为 ROW）
SELECT @@binlog_format;
-- 如果不是 ROW，需修改：
SET GLOBAL binlog_format = 'ROW';
```

## 量化对比

| 指标 | bad.sql（RR） | good.sql（RC） |
|------|--------------|---------------|
| 锁范围 | 索引区间（含间隙） | **仅命中的行** |
| 并发插入 | 被阻塞 | **不受影响** |
| 锁等待超时 | 常见 | **极少** |
| 系统吞吐量 | 低 | **高** |
| 幻读 | 防止 | 可能出现 |

## 5.7 vs 8.0 差异

- RC 和 RR 的锁行为在两个版本中一致
- 8.0 的 `performance_schema.data_locks` 可验证 RC 下确实没有间隙锁
- 8.0 默认隔离级别仍是 RR，需显式切换到 RC

::: tip 隔离级别选择建议
大多数互联网应用推荐使用 RC + ROW binlog，配合乐观锁（version）或悲观锁（FOR UPDATE）保证业务一致性。RC 的并发性能显著优于 RR，且幻读问题在大多数业务场景下可通过应用层逻辑规避。金融级一致性要求高的场景才需要使用 RR。
:::
