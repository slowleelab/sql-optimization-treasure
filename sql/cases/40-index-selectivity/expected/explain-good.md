# EXPLAIN 参考结果 - good.sql (联合索引提升选择性)

## MySQL 8.0（实测 8.0.46，20 万行数据）

```
+----+-------------+-----------------+------+----------------+----------------+---------+-------------+------+----------+-------+
| id | select_type | table           | type | possible_keys  | key            | key_len | ref         | rows | filtered | Extra |
+----+-------------+-----------------+------+----------------+----------------+---------+-------------+------+----------+-------+
|  1 | SIMPLE      | t_order_status  | ref  | idx_status_user| idx_status_user| 16      | const,const |    4 |   100.00 | NULL  |
+----+-------------+-----------------+------+----------------+----------------+---------+-------------+------+----------+-------+
```

## 关键改进

| 字段 | 值 | 分析 |
|------|-----|------|
| type | **`ref`** | **联合索引等值匹配** |
| key | `idx_status_user` | 使用联合索引 (status, user_id) |
| key_len | `16` | status(1+对齐) + user_id(8) |
| ref | `const,const` | 两个等值条件都用上 |
| rows | ~4 | **仅扫描 4 行！** |
| Extra | NULL | 无额外操作 |

## 为什么更好

`(status, user_id)` 联合索引把低基数的 `status` 与高基数的 `user_id` 组合：

1. **选择性大幅提升**：`status=1` 命中 10 万行，但 `status=1 AND user_id=12345` 只命中个位数行
2. **索引被真正使用**：复合选择性接近 1，优化器果断走索引
3. **扫描行数骤降**：从 19.8 万行降到 4 行

## 选择性计算

```sql
-- 联合索引的选择性
SELECT COUNT(DISTINCT status, user_id) / COUNT(*) AS combined_sel
FROM t_order_status;
-- 结果接近 1（高选择性）
```

| 索引 | 选择性 | 优化器是否使用 |
|------|--------|----------------|
| idx_status | ~0.000015 | 否（全表扫描） |
| idx_status_user | ~0.99 | 是（ref 匹配） |

## 量化对比

| 指标 | bad.sql (单列索引) | good.sql (联合索引) | 提升 |
|------|--------------------|---------------------|------|
| type | ALL | ref | 全表 → 索引查找 |
| rows | ~198,421 | ~4 | **减少 99.998%** |
| 扫描方式 | 顺序全表 | 索引定位 | 随机命中 |
| 耗时 | ~80 ms | < 1 ms | **数十倍** |

::: tip 低基数列索引设计原则
- 单独给低基数列（status、gender、is_deleted）建索引通常无意义
- 将低基数列作为联合索引**前导列**，后接高基数列，整体选择性由高基数列保证
- 低基数列适合做**前导列**是因为它常用于等值过滤，且能利用 ICP
- 若仅需统计各状态数量，考虑维护汇总表而非依赖索引
:::
