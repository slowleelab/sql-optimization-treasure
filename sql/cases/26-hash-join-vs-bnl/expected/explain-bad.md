# EXPLAIN 参考结果 - bad.sql（无索引 JOIN：5.7 BNL vs 8.0 Hash Join）

> 本案例 bad.sql 与 good.sql **SQL 完全相同**，差异在于是否执行了 `setup-good.sql`
> 为 `t_b.a_id` 建索引。bad 即未建索引时的执行计划，用于对比 5.7 BNL 与 8.0 Hash Join。

## 关键问题

| 字段 | 值 | 分析 |
|------|-----|------|
| 驱动表 `a` type | `ALL` | `t_a.val` 列**无索引**，`val > 49000` 走全表扫描 |
| 驱动表 `a` filtered | `2.00` | 5 万行中只有约 2%（约 1000 行）满足 `val > 49000`，但仍要全扫 5 万行 |
| 被驱动表 `b` type | `ALL` | `t_b.a_id` 列**无索引**，无法走 Index Nested Loop |
| 被驱动表 `b` key | `NULL` | 无可用索引 |
| 被驱动表 `b` rows | ~99,511 | 预估扫描约 10 万行（整表） |
| Extra | `Using join buffer (hash join)` | 8.0 用 Hash Join：对驱动表过滤后的结果建哈希表，再全表扫描被驱动表探测 |

## 为什么慢

两张表的 JOIN 列和过滤列都无索引，是典型的“无索引 JOIN”场景：

- **驱动表 `t_a`**：`val` 无索引，`WHERE a.val > 49000` 只能全表扫描约 5 万行，过滤出约 1000 行。
- **被驱动表 `t_b`**：`a_id` 无索引，无法对每行驱动记录做索引查找，必须用非索引路径完成连接。

**MySQL 8.0 Hash Join**：把驱动表过滤后的约 1000 行放入内存哈希表（key 为 `a.id`），然后**全表扫描 `t_b` 的约 10 万行**，逐行用 `b.a_id` 探测哈希表。复杂度 O(n+m)，只扫被驱动表一遍，比 5.7 的 BNL 高效，但仍需完整读一遍 10 万行表。

**MySQL 5.7 BNL**：更慢。把驱动表结果分块放入 join buffer，对**每一块**都全表扫描一次被驱动表。若 join buffer 容纳不下全部 1000 行驱动记录而分多块，`t_b` 会被重复全扫多遍，复杂度近似 O(n×m)。

根因：**被驱动表 JOIN 列 `t_b.a_id` 无索引**，无论用 Hash Join 还是 BNL 都要全表扫描被驱动表。

## MySQL 5.7 差异

5.7 不支持 Hash Join，无索引 JOIN 只能用 Block Nested Loop，被驱动表 `b` 的 Extra 显示 `Using join buffer (Block Nested Loop); Using where`，被驱动表被反复全表扫描，性能明显劣于 8.0。

|  1 | SIMPLE      | b     | NULL       | ALL  | NULL          | NULL |    NULL | NULL | 99511 |   100.00 | Using join buffer (Block Nested Loop); Using where |
