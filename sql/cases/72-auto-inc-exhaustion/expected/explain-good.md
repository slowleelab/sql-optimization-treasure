## 关键改进

| 指标 | bad.sql (INT) | good.sql (Snowflake BIGINT) | 分析 |
|------|---------------|------------------------------|------|
| 主键类型 | INT UNSIGNED | BIGINT | 上限从 42 亿 -> 9.2×10¹⁸ |
| ID 生成方式 | MySQL AUTO_INCREMENT | 应用层 Snowflake | 不依赖数据库自增 |
| 耗尽风险 | 3-5 年触发 | 69 年不触发 | 时间戳占 41 bit |
| 写入可用性 | 耗尽后中断 | 永不中断 | 彻底解决 |
| 分布式支持 | 不支持（需换步长） | 原生支持 | 10 bit 机器 ID |

## 为什么快且安全

雪花 ID 方案的核心优势：

1. **永不耗尽**：BIGINT 上限 9.2×10¹⁸，雪花算法每毫秒最多生成 4096 个 ID/机器，每年消耗约 3×10¹⁶，可用约 69 年
2. **应用层生成**：不依赖数据库 AUTO_INCREMENT，无需数据库分配 ID，减少一次交互
3. **天然分布式**：10 bit 机器 ID 支持 1024 台机器同时生成 ID，无需协调
4. **趋势递增**：时间戳在高位，ID 大致按时间递增，B+ 树插入不会导致页分裂
5. **无锁竞争**：每台机器独立生成，无 AUTO_INCREMENT 锁争用

## 量化对比

| 维度 | INT AUTO_INCREMENT | BIGINT Snowflake |
|------|-------------------|-------------------|
| 上限 | 42.9 亿 | 9.2 × 10¹⁸ |
| 预计耗尽 | 3-5 年 | 69 年 |
| 紧急扩容 | ALTER TABLE 锁表 | 不需要 |
| 分布式 | 需设不同起始值/步长 | 原生支持 |
| ID 生成位置 | 数据库 | 应用层 |
| 插入性能 | 受自增锁限制 | 无锁，更高吞吐 |

## 5.7 vs 8.0 差异

- 雪花 ID 方案与 MySQL 版本无关，5.7 和 8.0 均可使用
- 8.0 的 `AUTO_INCREMENT` 值持久化到 redo log，重启不丢失（5.7 可能丢失导致重复 ID）
- 8.0 可通过 `ALTER TABLE ... AUTO_INCREMENT = n` 修改水位，但大表仍有锁表风险
