# EXPLAIN 参考结果 - bad.sql (优化前)

## MySQL 8.0（实测 8.0.46，50 万行数据）

```
+----+-------------+--------------+------+---------------+------+---------+------+--------+----------+-----------------+
| id | select_type | table        | type | possible_keys | key  | key_len | ref  | rows   | filtered | Extra           |
+----+-------------+--------------+------+---------------+------+---------+------+--------+----------+-----------------+
|  1 | SIMPLE      | t_order_stat | ALL  | NULL          | NULL | NULL    | NULL | 498616 |   100.00 | Using temporary |
+----+-------------+--------------+------+---------------+------+---------+------+--------+----------+-----------------+
```

## 关键问题

| 字段 | 值 | 分析 |
|------|-----|------|
| type | `ALL` | 全表扫描 |
| key | NULL | 没用任何索引 |
| rows | ~498,616 | 扫描近 50 万行 |
| Extra | `Using temporary` | **需要创建临时表来完成分组** |

## 为什么慢

`GROUP BY city` 时 city 字段没有索引，MySQL 必须：
1. 全表扫描读取所有行
2. 创建临时表（Using temporary）存放分组中间结果
3. 对临时表做聚合计算

临时表可能落在磁盘上（数据量大时），I/O 开销巨大。

实际耗时：约 **424 ms**。
