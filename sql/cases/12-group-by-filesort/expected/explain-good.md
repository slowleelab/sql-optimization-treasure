# EXPLAIN 参考结果 - good.sql (优化后)

## MySQL 8.0（实测 8.0.46，50 万行数据）

```
+----+-------------+--------------+-------+---------------+----------+---------+------+--------+----------+-------+
| id | select_type | table        | type  | possible_keys | key      | key_len | ref  | rows   | filtered | Extra |
+----+-------------+--------------+-------+---------------+----------+---------+------+--------+----------+-------+
|  1 | SIMPLE      | t_order_stat | index | NULL          | idx_city | 82      | NULL | 498616 |   100.00 | NULL  |
+----+-------------+--------------+-------+---------------+----------+---------+------+--------+----------+-------+
```

## 关键改进

| 字段 | 值 | 分析 |
|------|-----|------|
| type | `index` | 索引全扫描（比 ALL 好，走索引有序） |
| key | `idx_city` | 使用了新建的索引 |
| Extra | NULL | **消除了 Using temporary！** |

## 为什么更好

加了 `idx_city` 索引后：
1. 索引本身有序 -> GROUP BY 不需要临时表排序
2. `Using temporary` 消失，无需创建中间临时表
3. 聚合计算直接在索引扫描过程中完成

## 量化对比

| 指标 | bad.sql | good.sql | 提升 |
|------|---------|----------|------|
| Extra | Using temporary | NULL | **消除临时表** |
| type | ALL | index | 从全表扫描变索引扫描 |

::: tip 注意
在全量 GROUP BY（无 WHERE 过滤）场景下，索引全扫描的绝对耗时可能不一定更快，因为仍需扫描全部行。但在有 WHERE 过滤条件时，索引可以缩小扫描范围，优势显著。本案例的核心价值是**消除 Using temporary**，避免大临时表落盘。
:::
