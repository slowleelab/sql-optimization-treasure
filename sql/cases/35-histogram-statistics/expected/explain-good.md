# EXPLAIN 参考结果 - good.sql（直方图让优化器估算更精确）

## MySQL 8.0（实测 8.0.46，20 万行数据，已创建 status 直方图）

执行 setup-good.sql 后，直方图记录了 status 的真实分布。可通过以下查询确认：
`SELECT TABLE_NAME, COLUMN_NAME FROM information_schema.COLUMN_STATISTICS WHERE TABLE_NAME='t_task';`

```
+----+-------------+--------+------+-----------------------------------+------------------+---------+-------+------+----------+-------------+
| id | select_type | table  | type | possible_keys                     | key              | key_len | ref   | rows | filtered | Extra       |
+----+-------------+--------+------+-----------------------------------+------------------+---------+-------+------+----------+-------------+
|  1 | SIMPLE      | t_task | ref  | idx_status,idx_user_created       | idx_user_created | 8       | const |  101 |   50.00  | Using where |
+----+-------------+--------+------+-----------------------------------+------------------+---------+-------+------+----------+-------------+
```

## 关键改进

| 字段 | 值 | 分析 |
|------|-----|------|
| type | `ref` | 索引等值查找 |
| key | `idx_user_created` | 优化器选择了正确的索引 |
| rows | ~101 | 通过 user_id 精确定位，仅约 100 行 |
| filtered | 50.00 | 回表后 status 过滤开销小 |

## 直方图的价值

直方图让优化器知道 `status=0` 实际占 99%（19.8 万行），而非均匀分布假设的约 33%。

在本例中，由于 `user_id=12345` 的选择性远高于 `status=0`，两个索引路径的代价差距巨大（10402 vs 35），优化器即使没有直方图也选对了索引。直方图的核心价值体现在以下场景：

1. **代价接近的边界场景**：当两个索引的预估代价相近时，直方图提供的精确分布信息能让优化器做出正确决策，避免因低估高倾斜值匹配行数而误选索引
2. **仅过滤高倾斜列**：如 `WHERE status = 0`（不叠加 user_id），直方图让优化器知道 status=0 占 99%，从而选择全表扫描而非索引扫描（全表顺序 I/O 比近全表索引回表的随机 I/O 更快）
3. **多列组合查询**：当 WHERE 涉及多个低基数列时，直方图帮优化器精确估算组合选择性

### 直方图前后的代价估算对比（JSON EXPLAIN）

| 指标 | 无直方图 | 有直方图 |
|------|---------|---------|
| idx_status rows_examined | ~99,937 | ~99,937（不变，但 filtered 更准） |
| idx_user_created query_cost | 35.35 | 0.35（统计更新后更精确） |
| 优化器对 status=0 的估值 | ~66,000（均匀假设） | ~198,000（直方图精确值） |

## 量化对比

| 指标 | bad.sql (无直方图) | good.sql (有直方图) | 说明 |
|------|--------------------|---------------------|------|
| key | idx_user_created | idx_user_created | 本例两者选择相同 |
| status=0 估值 | ~66,000 (失真) | ~198,000 (精确) | **直方图修正估值** |
| 边界场景选错风险 | 较高 | 极低 | 直方图消除选错风险 |

::: tip 直方图的核心价值
直方图不改变执行路径，而是让优化器**估算更精确**，从而在代价接近时做对决策。

1. **数据倾斜列**：status、type、is_deleted 等枚举列分布不均时效果显著
2. **索引选择两难**：当多个索引代价接近时，直方图帮优化器选出代价最低的
3. **零运行时开销**：直方图是统计信息，存储在 `information_schema.COLUMN_STATISTICS`，仅在优化阶段读取
4. **8.0 专属**：`ANALYZE TABLE ... UPDATE HISTOGRAM ON` 是 8.0 新特性，5.7 无此功能

注意：直方图不会修改索引或数据。对 `status=2`（仅占 0.5%）的查询，优化器知道它选择性高，会选择 `idx_status` 索引。直方图的价值是让优化器**对不同值做出不同最优决策**。
:::
