# EXPLAIN 参考结果 - bad.sql (优化前)

## MySQL 8.0（实测 8.0.46，100 万行数据）

```
+----+-------------+--------+------------+------+----------------------+----------------------+---------+-------+--------+----------+-----------------------+
| id | select_type | table  | partitions | type | possible_keys        | key                  | key_len | ref   | rows   | filtered | Extra                 |
+----+-------------+--------+------------+------+----------------------+----------------------+---------+-------+--------+----------+-----------------------+
|  1 | SIMPLE      | t_feed | NULL       | ref  | idx_status_created_id| idx_status_created_id| 1       | const | 666432 |   100.00 | Backward index scan   |
+----+-------------+--------+------------+------+----------------------+----------------------+---------+-------+--------+----------+-----------------------+
```

## 关键问题

| 字段 | 值 | 分析 |
|------|-----|------|
| type | `ref` | 走了索引定位 status=1 |
| key | `idx_status_created_id` | 用了 (status, created_at, id) 索引 |
| rows | ~666,432 | 预估扫描约 66 万行 |
| Extra | `Backward index scan` | 8.0 逆向索引扫描 |

## 为什么慢

虽然走了索引定位 `status=1` 的行，但 **LIMIT 1000000, 20** 要求 MySQL 扫描前 100 万行并丢弃，再返回 20 行。

实际行为：
1. 通过 `idx_status_created_id` 逆向扫描找到所有 `status=1` 的行
2. **逐行回表**读取完整数据（`SELECT *` 需要所有字段）
3. 跳过前 1,000,000 行（每一行都付出了回表代价）
4. 返回 20 行

**被丢弃的 100 万行也要回表**，而且越往后翻 OFFSET 越大，性能线性退化。

实际耗时：约 **520 ms**（实测 MySQL 8.0.46，100 万行数据，第 5 万页）。

## MySQL 5.7 差异

5.7 中 Extra 会显示 `Using filesort` 而非 `Backward index scan`，因为 5.7 不支持降序索引的逆向扫描优化。性能表现类似或略差。
