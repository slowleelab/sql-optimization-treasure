# EXPLAIN 参考结果 - good.sql（改写为范围查询避免函数）

## MySQL 8.0（30 万行数据）

```
+----+-------------+--------------+------------+-------+---------------+------------+---------+------+------+----------+-----------------------+
| id | select_type | table        | partitions | type  | possible_keys | key        | key_len | ref  | rows | filtered | Extra                 |
+----+-------------+--------------+------------+-------+---------------+------------+---------+------+------+----------+-----------------------+
|  1 | SIMPLE      | t_order_func | NULL       | range | idx_created   | idx_created| 5       | NULL |    3 |   100.00 | Using index condition |
+----+-------------+--------------+------------+-------+---------------+------------+---------+------+------+----------+-----------------------+
```

## 关键改进

| 字段 | 值 | 分析 |
|------|-----|------|
| type | `range` | 索引范围扫描 |
| key | `idx_created` | 索引被正确使用 |
| rows | `3` | 精确定位到 3 行 |
| Extra | `Using index condition` | 索引条件下推 |

## 为什么快

将 `DATE(created_at) = '2026-07-01'` 改写为 `created_at >= '2026-07-01' AND created_at < '2026-07-02'`，不再对索引列施加函数，优化器可以直接在 B+ 树上做范围查找，定位到 7 月 1 日的 3 条记录。
