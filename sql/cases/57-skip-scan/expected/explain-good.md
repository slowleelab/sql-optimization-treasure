# EXPLAIN 参考结果 - good.sql（显式展开前导列，索引完全生效）

> 本案例无 setup-good.sql，bad/good 差异在于 SQL 改写方式。

## MySQL 8.0（实测 8.0.46，50 万行数据）

```
+----+-------------+-------------+------------+-------+--------------------+--------------------+---------+------+--------+----------+--------------------------+
| id | select_type | table       | partitions | type  | possible_keys      | key                | key_len | ref  | rows   | filtered | Extra                    |
+----+-------------+-------------+------------+-------+--------------------+--------------------+---------+------+--------+----------+--------------------------+
|  1 | SIMPLE      | t_user_skip | NULL       | range | idx_gender_created | idx_gender_created | 10      | NULL | 166244 |   100.00 | Using where; Using index |
+----+-------------+-------------+------------+-------+--------------------+--------------------+---------+------+--------+----------+--------------------------+
```

## 关键改进

| 字段 | 值 | 分析 |
|------|-----|------|
| type | `range` | 索引范围扫描 |
| possible_keys | `idx_gender_created` | 联合索引可用 |
| key | `idx_gender_created` | 实际使用了联合索引 |
| key_len | `10` | gender (CHAR(1) utf8mb4 = 4字节) + created_at (DATETIME = 5字节) + 1字节NULL标记 = 10字节 |
| rows | ~166,244 | 预估扫描约 16.6 万行（约 1/3 数据） |
| filtered | 100.00% | 所有扫描行都满足条件 |
| Extra | `Using where; Using index` | 使用索引覆盖扫描，无需回表 |

## 为什么快

显式展开前导列 `gender IN ('M','F')` 后，联合索引 `(gender, created_at)` 完全生效：

1. MySQL 分别在 `gender='M'` 和 `gender='F'` 两个索引前缀下，利用 `created_at > '2026-01-01'` 做范围扫描
2. 每个前缀下的扫描都是高效的索引范围扫描（`type=range`）
3. `Using index` 表示覆盖索引扫描，无需回表读取数据行

与 bad 方案的关键差异：
- **bad**：跳过前导列，无法使用索引，全表扫描 50 万行
- **good**：显式展开前导列，索引范围扫描约 16.6 万行，且是覆盖索引扫描

扫描行数从 50 万降至 16.6 万，且避免了回表操作，性能显著提升。

实际耗时：约 **45 ms**。

## 量化对比

| 指标 | bad.sql | good.sql | 提升 |
|------|---------|----------|------|
| type | ALL | range | 全表扫描 → 索引范围扫描 |
| key | NULL | idx_gender_created | 索引生效 |
| rows | ~498,732 | ~166,244 | 减少 67% |
| Extra | Using where | Using where; Using index | 覆盖索引，无需回表 |
| 耗时 | ~180 ms | ~45 ms | **4 倍** |

## 5.7 vs 8.0 差异

- **5.7**：不支持 Skip Scan，bad 方案只能全表扫描；good 方案显式展开前导列后索引生效，提升显著
- **8.0**：支持 Skip Scan，bad 方案理论上可用 Skip Scan，但优化器选择不稳定；good 方案显式展开更可靠，执行计划更稳定
- 显式展开前导列是两版通用的最佳实践，不依赖优化器的 Skip Scan 决策
