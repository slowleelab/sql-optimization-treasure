# EXPLAIN 参考结果 - good.sql (优化后)

## MySQL 8.0（实测 8.0.46，约 30 万行数据）

```
+----+-------------+-----------+------+-------------------+-------------------+---------+-------+------+----------+-------------+
| id | select_type | table     | type | possible_keys     | key               | key_len | ref   | rows | filtered | Extra       |
+----+-------------+-----------+------+-------------------+-------------------+---------+-------+------+----------+-------------+
|  1 | SIMPLE      | t_product | ref  | idx_category_price| idx_category_price| 4       | const | 2987 |   100.00 | Using index |
+----+-------------+-----------+------+-------------------+-------------------+---------+-------+------+----------+-------------+
```

## 关键改进

| 字段 | 值 | 分析 |
|------|-----|------|
| type | `ref` | 与 bad 相同 |
| key | `idx_category_price` | 与 bad 相同 |
| rows | 2987 | 与 bad 相同 |
| Extra | **`Using index`** | **覆盖索引！不回表！** |

## 为什么快

查询改为 `SELECT id, category_id, price`，这三个字段全部包含在索引中：

- `category_id` -> 索引的第一列
- `price` -> 索引的第二列
- `id` -> InnoDB 二级索引自动附加主键

因此 MySQL 可以**完全在索引上完成查询**，不需要回表到聚簇索引。

## 量化对比

| 指标 | bad.sql | good.sql | 提升 |
|------|---------|----------|------|
| Extra | NULL（需回表） | **Using index** | **0 次回表** |
| 回表次数 | 2987 | **0** | 完全消除 |
| I/O 模式 | 随机 I/O（回表） | 顺序 I/O（索引扫描） | 快数倍 |

::: tip 覆盖索引判定
查询的所有列都在索引中时，Extra 会显示 `Using index`。对于 InnoDB，二级索引自动包含主键，因此 `SELECT id` 总是被覆盖的。
:::
