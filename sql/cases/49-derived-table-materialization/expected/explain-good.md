# EXPLAIN 参考结果 - good.sql (直接 GROUP BY HAVING)

## MySQL 8.0（20 万行访问日志，5000 个用户分组）

```
+----+-------------+---------------+------------+-------+---------------+---------------+---------+------+--------+----------+----------------------------------------------+
| id | select_type | table         | partitions | type  | possible_keys | key           | key_len | ref  | rows   | filtered | Extra                                        |
+----+-------------+---------------+------------+-------+---------------+---------------+---------+------+--------+----------+----------------------------------------------+
|  1 | SIMPLE      | t_access_log  | NULL       | index | idx_user_id   | idx_user_id   | 8       | NULL | 199430 |   100.00 | NULL                                         |
+----+-------------+---------------+------------+-------+---------------+---------------+---------+------+--------+----------+----------------------------------------------+
```

## MySQL 5.7（20 万行访问日志，5000 个用户分组）

```
+----+-------------+---------------+------------+-------+---------------+---------------+---------+------+--------+----------+----------------+
| id | select_type | table         | partitions | type  | possible_keys | key           | key_len | ref  | rows   | filtered | Extra          |
+----+-------------+---------------+------------+-------+---------------+---------------+---------+------+--------+----------+----------------+
|  1 | SIMPLE      | t_access_log  | NULL       | index | idx_user_id   | idx_user_id   | 8       | NULL | 199430 |   100.00 | Using filesort |
+----+-------------+---------------+------------+-------+---------------+---------------+---------+------+--------+----------+----------------+
```

## 关键改进

| 字段 | 值 | 分析 |
|------|-----|------|
| type | `index` | 扫描 idx_user_id 索引（GROUP BY 利用有序性） |
| key | `idx_user_id` | 索引有序扫描，GROUP BY 无需临时表 |
| rows | ~199,430 | 扫描全部日志行 |
| Extra (8.0) | `NULL` | **无派生表物化，无临时表** |
| HAVING | 直接过滤 | 聚合时直接过滤，只产出满足条件的行 |
| 结果行数 | ~1600 | 只输出 cnt>100 的用户（vs bad 物化 5000 行） |

## 为什么快

1. **消除派生表**：不再有 FROM 子查询，单层 SIMPLE 查询
2. **HAVING 直接过滤**：`HAVING COUNT(*) > 100` 在聚合时直接判断，不产出不满足条件的行
3. **索引有序 GROUP BY**：idx_user_id 有序扫描，GROUP BY 无需临时表（Extra 为 NULL）
4. **无物化开销**：不创建 DERIVED 临时表，省去物化 + 外层扫描的代价

对比 bad 方案物化 5000 行再外层过滤 -> good 方案聚合时直接过滤，只产出约 1600 行。

实际耗时：
- 5.7：约 **310 ms**（无物化，但 5.7 GROUP BY 仍有 Using filesort）
- 8.0：约 **260 ms**（索引有序，无 filesort，无物化）

## 量化对比

| 指标 | bad.sql (5.7) | bad.sql (8.0) | good.sql (5.7) | good.sql (8.0) |
|------|--------------|--------------|----------------|----------------|
| 耗时 | 420 ms | 380 ms | 310 ms | **260 ms** |
| 派生表物化 | 是(5000行) | 部分 | **无** | **无** |
| 临时表 | 有 | 可能 | **无** | **无** |
| filesort | 有 | 有 | 有 | **无** |
| 提升(vs 5.7 bad) | - | - | 1.35x | **1.6x** |

## 5.7 vs 8.0 差异

- **5.7**：派生表总是物化，good 方案消除物化后提升明显；但 5.7 GROUP BY 仍可能 Using filesort
- **8.0**：派生表有条件下推优化，但 good 方案直接绕过派生表，两版本都受益
- **8.0 索引有序 GROUP BY**：idx_user_id 有序扫描时，8.0 可完全消除 filesort

::: tip 优化原则
避免用派生表(FROM 子查询)包裹 GROUP BY 再外层过滤。聚合过滤应直接用 HAVING，让优化器在聚合阶段就过滤，而非先物化全部结果再过滤。8.0 的条件下推是兜底优化，但不能替代正确的 SQL 写法。当分组基数大（百万级 user_id）时，物化开销会放大，good 方案的提升更显著。
:::
