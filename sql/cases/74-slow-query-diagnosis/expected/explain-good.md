## 关键改进

### SQL 1 改进

| 字段 | 优化前 | 优化后 | 分析 |
|------|--------|--------|------|
| type | `ALL` | `ref` + `eq_ref` | 子查询走索引定位，外层主键关联 |
| key | `NULL` | `idx_status_created_id` | 用上 (status, created_at, id) 三列索引 |
| rows | ~498,512 | 子查询 ~125,375 / 外层 20 | 子查询走索引有序扫描，外层仅回表 20 次 |
| Extra | `Using where; Using filesort` | `Backward index scan` | 索引天然有序，消除 filesort |

### SQL 2 改进

| 字段 | 优化前 | 优化后 | 分析 |
|------|--------|--------|------|
| key | `idx_user` | `idx_user_amount` | 复合索引含 amount |
| Extra | `Using filesort` | `Using index condition` | amount 在索引中有序，消除 filesort |

### SQL 3 改进

| 字段 | 优化前 | 优化后 | 分析 |
|------|--------|--------|------|
| filtered | `100.00` | `50.00` | 范围条件让优化器预估过滤掉一半（条件更精确） |
| Extra | `Using where`（含逐行 DATE()） | `Using where` | 消除逐行 DATE() 函数调用 |

## 为什么快

**SQL 1（延迟关联 + 三列索引）**：
- `(status, created_at, id)` 索引同时满足 `WHERE status=1` 过滤 + `ORDER BY created_at DESC` 排序，索引有序无需 filesort
- 子查询 `SELECT id` 只取主键，走**覆盖索引**（索引含 id），不回表就定位到目标 20 条的 id
- 外层 `JOIN tmp.id ON t.id = tmp.id` 用主键 `eq_ref` 只回表 20 次
- bad 方案回表 ~100,020 次；good 方案回表 20 次
- 实际耗时：约 **6 ms**（从 820 ms 降到 6 ms）

**SQL 2（复合索引 idx_user_amount）**：
- `WHERE user_id=12345` 走索引第一列等值匹配
- `ORDER BY amount DESC` 用索引第二列的有序性，**无需 filesort**
- 消除了"回表读 amount 再排序"的环节
- 实际耗时：约 **1 ms**（从 15 ms 降到 1 ms）

**SQL 3（范围查询改写）**：
- `created_at >= '2026-07-01' AND created_at < '2026-07-02'` 让列保持"裸"状态
- 消除逐行调用 `DATE()` 函数的开销
- 若配合 `idx_created` 索引，可走 `range` 扫描只读当天数据（本案例聚焦诊断链路未单独建该索引）
- 实际耗时：约 **180 ms**（从 480 ms 降到 180 ms，消除函数开销；建索引后可达 ms 级）

## 诊断链路价值小结

本案例的真正价值不在"单条 SQL 优化技巧"，而在于**用一套方法论从海量慢查询中精准定位 TOP N**：

1. **slow log 采集**：把超过阈值的 SQL 落盘，是所有诊断的原始数据
2. **pt-query-digest 聚合**：把数万条日志按指纹聚合成几十类，按总耗时排序，TOP N 一眼可见
3. **performance_schema 实时统计**：无需等日志，实时看 SQL 累计耗时，与 pt-query-digest 交叉验证
4. **EXPLAIN / EXPLAIN ANALYZE 验证**：对 TOP N 逐条看执行计划，定位慢因（全表扫描 / filesort / 函数失效）
5. **逐条治理**：建索引 + 改 SQL，3 条慢 SQL 优化后整体 CPU 下降 60%

没有这套链路，DBA 只能看到"CPU 高"却无从下手；有了它，85% 的问题能在 1 小时内定位。

## 5.7 vs 8.0 差异

- 执行计划结构基本一致，8.0 显示 `Backward index scan`（索引逆向扫描），5.7 显示 `Using filesort`（需额外排序）
- **EXPLAIN ANALYZE 是 8.0 独有**：能输出实际行数和耗时，验证优化器预估是否准确；5.7 只能看预估值
- performance_schema：8.0 默认开启且 digest 采集完善；5.7 需手动开启并配置 instruments/consumers
- 索引方案（idx_status_created_id、idx_user_amount）和 SQL 改写在两个版本上均有效
