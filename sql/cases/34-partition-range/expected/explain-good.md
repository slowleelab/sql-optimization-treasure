# EXPLAIN 参考结果 - good.sql (分区表分区裁剪)

## MySQL 8.0（实测 8.0.46，96 万行日志数据，12 个月度分区）

```
+----+-------------+------------------+------------+-------+---------------+-------------+---------+------+--------+----------+-------------+
| id | select_type | table            | partitions | type  | possible_keys | key         | key_len | ref  | rows   | filtered | Extra       |
+----+-------------+------------------+------------+-------+---------------+-------------+---------+------+--------+----------+-------------+
|  1 | SIMPLE      | t_partition_log  | p202401    | range | idx_created   | idx_created | 6       | NULL |  78520 |   100.00 | Using where |
+----+-------------+------------------+------------+-------+---------------+-------------+---------+------+--------+----------+-------------+
```

（注意 partitions 列显示 `p202401`，而非 NULL 或全部 12 个分区）

## 关键改进

| 字段 | 值 | 分析 |
|------|-----|------|
| partitions | **`p202401`** | **分区裁剪！只访问 1 月分区** |
| type | `range` | 走 idx_created 索引范围扫描 |
| key | `idx_created` | 分区内的 created_at 索引 |
| rows | ~78,520 | 预估扫描 1 月约 7.8 万行 |
| Extra | `Using where` | 索引范围扫描后过滤 |

## 为什么快

1. **分区裁剪**：优化器根据 `created_at BETWEEN '2024-01-01' AND '2024-01-31'` 判断只需访问 p202401 分区，跳过其他 11 个分区
2. **索引物理隔离**：每个分区有独立的索引 B+ 树，p202401 的 idx_created 只含 8 万行，索引更紧凑，树高更低
3. **数据物理隔离**：1 月数据集中在 p202401 分区的数据页中，buffer pool 局部性好
4. **EXPLAIN 直接可见**：partitions 列从 NULL 变为 p202401，裁剪效果一目了然

**管理成本优势**：
- 清理 1 月数据：`ALTER TABLE t_partition_log DROP PARTITION p202401`（瞬间完成，无大事务）
- 各分区可独立维护，DDL 影响范围小

实际耗时：约 **95 ms**（实测 MySQL 8.0.46，96 万行数据）。

## 量化对比

| 指标 | bad.sql (普通表) | good.sql (分区表) | 提升 |
|------|-----------------|------------------|------|
| 耗时 | 180 ms | 95 ms | **约 1.9 倍** |
| partitions 列 | NULL | **p202401** | 分区裁剪生效 |
| 扫描分区数 | 1（全表） | **1（仅 p202401）** | 物理隔离 |
| 索引树高度 | 较高(96万行) | 较低(8万行) | 更紧凑 |
| 历史数据清理 | DELETE(大事务) | DROP PARTITION(瞬间) | **数量级提升** |

随着数据量和分区数增长，分区裁剪的优势会更明显：
- 1000 万行 / 12 分区 -> 每分区约 83 万行，裁剪后扫 1/12
- 查询跨多月时，partitions 显示多个分区（如 p202401,p202402）

## 5.7 vs 8.0 差异

- 5.7 和 8.0 都支持 RANGE 分区和分区裁剪，核心机制一致
- 8.0 的分区裁剪优化器更智能，对函数表达式分区键的裁剪判断更准确
- 8.0 支持 `ALGORITHM=INPLACE` 的分区维护操作，DDL 影响更小
- 两版本的 `EXPLAIN ... PARTITIONS` 语法一致，partitions 列行为相同

::: tip 分区表设计要点
1. 分区键必须包含在主键/唯一键中（`PRIMARY KEY (id, created_at)`）
2. RANGE 分区适合按时间范围查询和清理的场景
3. 分区裁剪依赖 WHERE 条件中包含分区键，且不能对分区键施加函数（除非用 TO_DAYS 等分区函数一致的表达式）
4. 分区数不宜过多（建议 < 1000），过多分区会增加元数据管理开销
5. 清理历史数据优先用 `DROP PARTITION` 而非 `DELETE`，避免大事务和碎片
:::
