# EXPLAIN 参考结果 - good.sql (分批更新，每次 1000 行)

## MySQL 8.0（100 万行订单数据，约 50 万行待更新）

```
-- EXPLAIN UPDATE t_order SET status = 3 WHERE status = 2 AND created_at < '2026-01-01' LIMIT 1000;
+----+-------------+---------+------------+-------+--------------------------+------------------+---------+------+--------+----------+-------------+
| id | select_type | table   | partitions | type  | possible_keys            | key              | key_len | ref  | rows   | filtered | Extra       |
+----+-------------+---------+------------+-------+--------------------------+------------------+---------+------+--------+----------+-------------+
|  1 | UPDATE      | t_order | NULL       | range | idx_status_created       | idx_status_created | 6     | NULL | 499830 |   100.00 | Using where |
+----+-------------+---------+------------+-------+--------------------------+------------------+---------+------+--------+----------+-------------+
```

执行计划与 bad.sql 相同（都是索引范围扫描），**优化点不在于执行计划，而在于分批执行减少锁持有时间**。

## 关键改进

| 维度 | bad.sql（一次性） | good.sql（分批） |
|------|------------------|-----------------|
| 每次更新行数 | ~500,000 | ~1,000 |
| 锁持有时间 | ~30 秒 | ~60 ms |
| undo log 累积 | 50 万条 | 1000 条/批 |
| 主从延迟 | +30 秒 | 几乎无影响 |
| 并发阻塞 | 严重 | 几乎无感知 |

## 为什么快

### 分批 UPDATE 的执行流程

1. 第 1 批：更新 1000 行，提交，释放锁（耗时 ~60 ms）
2. 第 2 批：更新 1000 行，提交，释放锁（耗时 ~60 ms）
3. ...
4. 第 500 批：更新最后 1000 行，提交，释放锁

**核心优势**：
- 每批只锁 1000 行，锁持有时间从 30 秒降到 60 ms
- 每批提交后 undo log 及时 purge，不会膨胀
- 主从延迟可控（每批在从库回放仅 60 ms）
- 并发事务几乎不会感知到锁等待

### 分批执行时间线

```
时间线   会话A（分批 UPDATE）                    会话B（不受影响）
  T1     BEGIN;
  T2     UPDATE ... LIMIT 1000;  -- 锁定 1000 行
  T3     COMMIT;                 -- 释放锁（60 ms）
  T4                                          UPDATE ... WHERE id=500000;
                                              -- 正常执行，无阻塞
  T5     BEGIN;
  T6     UPDATE ... LIMIT 1000;  -- 锁定下一批 1000 行
  T7     COMMIT;                 -- 释放锁（60 ms）
  ...
         => 会话B 在 T4 时刻正常执行，不受会话A 影响
```

### 分批大小选择

| 分批大小 | 每批耗时 | 总批次 | 总耗时 | 适用场景 |
|---------|---------|-------|-------|---------|
| 100 | ~6 ms | 5000 | ~30 秒 | 极高并发 |
| 500 | ~30 ms | 1000 | ~30 秒 | 高并发 |
| 1000 | ~60 ms | 500 | ~30 秒 | 通用（推荐） |
| 5000 | ~300 ms | 100 | ~30 秒 | 低并发 |
| 10000 | ~600 ms | 50 | ~30 秒 | 极低并发 |

**推荐**：1000 行/批是经验值，可根据业务并发量调整。并发越高，分批越小。

### 应用层实现

```java
// Java 示例：分批更新
public void batchUpdate() {
    int affected;
    do {
        affected = jdbcTemplate.update(
            "UPDATE t_order SET status = 3 WHERE status = 2 AND created_at < '2026-01-01' LIMIT 1000"
        );
        if (affected > 0) {
            Thread.sleep(10);  // 可选：短暂休眠，降低对线上业务的影响
        }
    } while (affected > 0);
}
```

```sql
-- 存储过程示例：分批更新
DELIMITER $$
CREATE PROCEDURE sp_batch_update()
BEGIN
    DECLARE v_affected INT DEFAULT 1;
    WHILE v_affected > 0 DO
        UPDATE t_order SET status = 3 WHERE status = 2 AND created_at < '2026-01-01' LIMIT 1000;
        SET v_affected = ROW_COUNT();
        COMMIT;
        DO SLEEP(0.01);  -- 休眠 10ms
    END WHILE;
END $$
DELIMITER ;
```

## 量化对比

| 指标 | bad.sql（一次性） | good.sql（分批） |
|------|------------------|-----------------|
| 每次更新行数 | ~500,000 | **~1,000** |
| 锁持有时间 | ~30 秒 | **~60 ms** |
| undo log 累积 | 50 万条 | **1000 条/批** |
| 主从延迟 | +30 秒 | **几乎无影响** |
| 并发阻塞 | 严重 | **几乎无感知** |
| 总耗时 | ~30 秒 | ~30 秒（相同） |

**注意**：分批更新的总耗时与一次性更新相同（都是 30 秒），但锁持有时间和系统影响大幅降低。

## 5.7 vs 8.0 差异

- 分批更新在两个版本中同样有效
- 8.0 的 undo log 管理更高效，分批更新下 purge 更及时
- 8.0 的 `performance_schema.data_locks` 可验证每批只锁 1000 行

::: tip 大批量 DML 优化原则
1. **分批执行**：大批量 UPDATE/DELETE 分批执行，每批 500~5000 行
2. **利用索引**：确保 WHERE 条件走索引，避免每批都全表扫描
3. **短暂休眠**：每批之间可加入 10~100ms 休眠，进一步降低对线上业务的影响
4. **监控主从延迟**：分批执行期间监控主从延迟，确保在可接受范围内
5. **低峰期执行**：大批量 DML 尽量在业务低峰期执行
:::
