## 关键改进

| 维度 | bad (无防护) | good (布隆过滤器) | 分析 |
|------|-------------|-------------------|------|
| 无效请求到达数据库 | 100% | **<1%（仅误判）** | 拦截 99%+ 无效请求 |
| 数据库 QPS | 5000（攻击 QPS） | **~50（误判率 1%）** | 降低 99% |
| 存在性判断位置 | 数据库 B+ 树 | **内存位数组** | 纳秒级 vs 微秒级 |
| 内存开销 | 0 | **~1.2 MB（100万ID）** | 极小内存换极大保护 |

## 为什么快

布隆过滤器的核心原理：

1. **位数组 + 多个哈希函数**：将每个存在的 ID 通过 k 个哈希函数映射到位数组的 k 个位置，置为 1
2. **查询时检查 k 个位**：如果 k 个位都是 1，则"可能存在"；如果任何一位是 0，则"一定不存在"
3. **"一定不存在"是确定的**：布隆过滤器说不存在，就一定不存在，可以直接返回
4. **"可能存在"有误判**：布隆过滤器说存在，可能实际不存在（误判率约 1%），需要查数据库确认

### 布隆过滤器工作流程

```
初始化:
  位数组: [0][0][0][0][0][0][0][0]  (全为0)

写入 id=100:
  hash1(100)=2, hash2(100)=5, hash3(100)=7
  位数组: [0][0][1][0][0][1][0][1]

写入 id=200:
  hash1(200)=1, hash2(200)=5, hash3(200)=3
  位数组: [0][1][1][1][0][1][0][1]

查询 id=300:
  hash1(300)=0, hash2(300)=5, hash3(300)=7
  位[0]=0 -> 一定不存在 -> 直接返回，不查数据库

查询 id=100:
  hash1(100)=2, hash2(100)=5, hash3(100)=7
  位[2]=1, 位[5]=1, 位[7]=1 -> 可能存在 -> 查数据库确认
```

### 防护效果对比

```
无防护（bad）:
  攻击 QPS 5000 -> 数据库 QPS 5000 -> 数据库被打爆

布隆过滤器（good）:
  攻击 QPS 5000
    -> 布隆过滤器拦截 4950（99%）-> 直接返回空
    -> 误判 50（1%）-> 查数据库 -> 返回空
  数据库 QPS: 50 -> 完全无压力
```

## 缓存穿透的完整防护方案

| 方案 | 原理 | 优点 | 缺点 |
|------|------|------|------|
| 布隆过滤器 | 内存位数组预判存在性 | 内存小、速度快 | 有误判率、不支持删除 |
| 缓存空结果 | 数据库 miss 也缓存（短 TTL） | 简单 | 占用缓存空间、TTL 内不一致 |
| 接口层校验 | 参数合法性检查（ID>0） | 零成本 | 只能拦截明显非法的 |
| 限流 | 对异常 IP 限流 | 通用 | 可能影响正常用户 |

## 5.7 vs 8.0 差异

- 布隆过滤器是应用层/中间件层的方案，与 MySQL 版本无关
- 8.0 的 Performance Schema 可更好地监控无效查询的来源
- 核心价值在于架构层面的防护设计

## 避坑指南

1. **布隆过滤器 + 缓存空结果组合使用**：布隆过滤器拦截大部分，缓存空结果兜底误判部分
2. **布隆过滤器要预热**：系统启动时将全量 ID 加载到布隆过滤器，新写入的 ID 也要同步添加
3. **误判率与内存的权衡**：误判率越低需要的内存越大，1% 误判率约需 9.6 bits/元素
4. **不支持删除**：标准布隆过滤器不支持删除操作，需要定期重建或使用计数布隆过滤器
5. **Redis 实现**：生产环境推荐用 Redis 的 `BF.ADD` / `BF.EXISTS` 命令（RedisBloom 模块）
6. **监控误判率**：定期统计布隆过滤器的误判率，超过阈值时考虑扩容位数组
