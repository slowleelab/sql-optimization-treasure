# EXPLAIN 参考结果 - bad.sql (缓存穿透，直接查库)

## MySQL 8.0（实测 8.0.46，用户表 100 万行）

```
+----+-------------+-------+------------+-------+---------------+---------+---------+-------+------+----------+-------+
| id | select_type | table | partitions | type  | possible_keys | key     | key_len | ref   | rows | filtered | Extra |
+----+-------------+-------+------------+-------+---------------+---------+---------+-------+------+----------+-------+
|  1 | SIMPLE      | NULL  | NULL       | NULL  | NULL          | NULL    | NULL    | NULL  | NULL | NULL     | no matching row in const table |
+----+-------------+-------+------------+-------+---------------+---------+---------+-------+------+----------+-------+
```

## 关键问题

| 字段 | 值 | 分析 |
|------|-----|------|
| type | `NULL` | 主键等值查询，优化器发现无匹配行 |
| Extra | `no matching row in const table` | 主键查找未命中 |
| 单次耗时 | ~0.1 ms | 单次查询本身很快 |

## 为什么慢

单看 EXPLAIN，这条查询本身极快——主键等值查找，发现不存在直接返回。真正的代价在**架构层面**：

1. **每次都打到数据库**：不存在的 ID 永远不会被缓存（缓存空结果策略未启用时），每次请求都穿透到数据库
2. **QPS 放大效应**：攻击者用 1 万个不存在的 ID，每秒每个 ID 查 1 次，数据库每秒承受 1 万次无效查询
3. **连接池耗尽**：大量无效查询占用数据库连接，正常业务查询排队甚至被拒绝
4. **CPU 空转**：每次查询都消耗 CPU 做 B+ 树查找，虽然单次快，但高并发下累积可观
5. **无法通过加索引解决**：主键查找已经是最快的查询方式，问题不在 SQL 本身

### 缓存穿透的攻击原理

```
正常请求（存在的 ID）:
  请求 -> 缓存(hit) -> 返回        <- 数据库无压力
  请求 -> 缓存(miss) -> 数据库 -> 写入缓存 -> 返回

恶意请求（不存在的 ID）:
  请求 -> 缓存(miss) -> 数据库(miss) -> 返回空
  请求 -> 缓存(miss) -> 数据库(miss) -> 返回空   <- 每次都打数据库
  请求 -> 缓存(miss) -> 数据库(miss) -> 返回空   <- 缓存永远 miss
  ...每秒数千次，数据库被打爆
```

### 生产环境的真实场景

```
无防护:
  攻击 QPS: 5000（全部是不存在的 ID）
  数据库 QPS: 5000（全部穿透到数据库）
  数据库连接: 被占满，正常业务查询超时
  结果: 数据库 CPU 100%，服务不可用

有布隆过滤器:
  攻击 QPS: 5000
  布隆过滤器拦截: 4950（99% 被拦截）
  数据库 QPS: 50（仅误判的 1% 到达数据库）
  结果: 数据库无感知，服务正常
```

实际耗时：单次约 **0.1 ms**，但每秒 5000 次无效查询累积导致数据库 CPU 飙升。

## MySQL 5.7 差异

5.7 行为一致。缓存穿透是架构层面的问题，与 MySQL 版本无关。
