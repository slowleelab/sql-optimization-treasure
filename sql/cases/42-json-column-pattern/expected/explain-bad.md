# EXPLAIN 参考结果 - bad.sql (直接 JSON_EXTRACT 查询)

## MySQL 8.0（t_product_json，10 万行，attrs 为 JSON 列）

```
+----+-------------+-----------------+------------+------+---------------+------+---------+------+--------+----------+-------------+
| id | select_type | table           | partitions | type | possible_keys | key  | key_len | ref  | rows   | filtered | Extra       |
+----+-------------+-----------------+------------+------+---------------+------+---------+------+--------+----------+-------------+
|  1 | SIMPLE      | t_product_json  | NULL       | ALL  | NULL          | NULL | NULL    | NULL | 100000 |   100.00 | Using where |
+----+-------------+-----------------+------------+------+---------------+------+---------+------+--------+----------+-------------+
```

## 关键问题

| 字段 | 值 | 分析 |
|------|-----|------|
| type | `ALL` | **全表扫描**！没有任何索引可用 |
| possible_keys | `NULL` | 优化器找不到可用索引 |
| key | `NULL` | 未使用任何索引 |
| rows | ~100,000 | 预估扫描全部 10 万行 |
| Extra | `Using where` | 逐行解析 JSON 后在 server 层过滤 |

## 为什么慢

1. **JSON 函数无法走索引**：`JSON_EXTRACT(attrs, '$.color')` 是函数表达式，MySQL 无法直接在 JSON 列 `attrs` 上用 B+ 树索引定位
2. **逐行解析 JSON**：每一行都要调用 JSON 解析器提取 `$.color`，再与 `'red'` 比较，CPU 开销大
3. **全表扫描**：10 万行全部扫描，即使只有约 1.6 万行（1/6）匹配 color='red'，也必须读完所有行
4. **SELECT \* 回表且全读**：所有行的完整数据都要读取，I/O 代价高
5. **数据量大时线性恶化**：100 万行就要扫描 100 万行，1000 万行就扫描 1000 万行，无法靠索引缩小范围

### JSON 字段查询的常见误区

```
误区: "JSON 字段灵活方便，直接查就行"
现实: JSON 字段本身不可索引，直接查 JSON 内部键 = 全表扫描

以下写法都会全表扫描:
  WHERE JSON_EXTRACT(attrs, '$.color') = 'red'
  WHERE attrs->'$.color' = 'red'        -- 语法糖，等价于上面
  WHERE attrs->>'$.color' = 'red'       -- 带去引号，仍全表扫描（无虚拟列索引时）

可行方案: 把高频查询的 JSON 键提取为虚拟列并建索引
```

实际耗时：约 **45 ms**（实测 MySQL 8.0.46，10 万行表）。
百万级数据表上同类查询可达 **数百毫秒甚至秒级**。
