# EXPLAIN 参考结果 - bad.sql (优化前)

## MySQL 8.0（实测 8.0.46，大表 20 万行 + 小表 5000 行）

```
+----+-------------+-------+------+---------------------+--------------+---------+-------------------------+--------+----------+-------------+
| id | select_type | table | type | possible_keys       | key          | key_len | ref                     | rows   | filtered | Extra       |
+----+-------------+-------+------+---------------------+--------------+---------+-------------------------+--------+----------+-------------+
|  1 | SIMPLE      | o     | ALL  | idx_order_no        | NULL         | NULL    | NULL                    | 199430 |   100.00 | NULL        |
|  1 | SIMPLE      | p     | ref  | idx_promotion,idx_order_no | idx_order_no | 130 | sql_treasure.o.order_no | 1 | 9.58 | Using where |
+----+-------------+-------+------+---------------------+--------------+---------+-------------------------+--------+----------+-------------+
```

（使用 STRAIGHT_JOIN 强制大表 t_order_big 在前作为驱动表）

## 关键问题

| 字段 | 值 | 分析 |
|------|-----|------|
| 驱动表 o | `type=ALL` | **大表全表扫描 20 万行** |
| 驱动表 rows | 199,430 | 外层循环 20 万次 |
| 被驱动表 p | `type=ref` | 用 idx_order_no 索引查找 |
| 被驱动表 filtered | 9.58% | 还需 WHERE 过滤 promotion_id |

## 为什么慢

大表做驱动表，外层循环 20 万次。每次去小表通过 idx_order_no 查找后还需用 `promotion_id=1` 过滤（filtered=9.58%），大量无效查找。

实际耗时：约 **4936 ms**。
