# EXPLAIN 参考结果 - bad.sql（STRAIGHT_JOIN 强制大表→中表→小表的最差顺序）

> 本案例无 setup-good.sql，bad/good 差异在于 `STRAIGHT_JOIN` 指定的 JOIN 顺序。
> bad 强制 `t_large → t_medium → t_small`（大表驱动），中间结果集巨大。
> 注意：bad 选 `s.*`，good 选 `l.*`，返回列不同。

## MySQL 8.0（t_small 1 千行 + t_medium 5 万行 + t_large 20 万行）

```
+----+-------------+-------+------------+--------+---------------+---------+---------+------------------+--------+----------+-------------+
| id | select_type | table | partitions | type   | possible_keys | key     | key_len | ref              |   rows | filtered | Extra       |
+----+-------------+-------+------------+--------+---------------+---------+---------+------------------+--------+----------+-------------+
|  1 | SIMPLE      | l     | NULL       | ALL    | NULL          | NULL    |    NULL | NULL             | 199723 |   100.00 | NULL        |
|  1 | SIMPLE      | m     | NULL       | eq_ref | PRIMARY       | PRIMARY |       8 | test.l.medium_id |      1 |   100.00 | NULL        |
|  1 | SIMPLE      | s     | NULL       | eq_ref | PRIMARY       | PRIMARY |       8 | test.m.small_id  |      1 |    10.00 | Using where |
+----+-------------+-------+------------+--------+---------------+---------+---------+------------------+--------+----------+-------------+
```

## 关键问题

| 字段 | 值 | 分析 |
|------|-----|------|
| 第一张表 `l` type | `ALL` | `t_large`（20 万行）做驱动表，**全表扫描**，无 WHERE 过滤（过滤条件在小表 `s` 上） |
| 第一张表 `l` rows | ~199,723 | 预估扫描约 20 万行，全部进入 JOIN 流水线 |
| 第二张表 `m` type | `eq_ref` | `m.id = l.medium_id` 走主键等值查找，每行 1 次 |
| 第三张表 `s` type | `eq_ref` | `s.id = m.small_id` 走主键等值查找，每行 1 次 |
| 第三张表 `s` filtered | `10.00` | `s.val = 1` 的过滤条件放在**最后一张表**才生效，约 10% 命中 |
| JOIN 顺序 | large → medium → small | 大表做驱动，中间结果集约 20 万行一路传递，最后才过滤 |

## 为什么慢

`STRAIGHT_JOIN` 强制了 `t_large → t_medium → t_small` 的顺序，把最大的表放在最前面当驱动表：

1. 先全表扫描 `t_large` 的约 20 万行，得到约 20 万行的中间结果集。
2. 对这 20 万行的每一行，到 `t_medium` 用主键查 1 行（`eq_ref`），中间结果仍约 20 万行。
3. 再对这 20 万行的每一行，到 `t_small` 用主键查 1 行（`eq_ref`），**此时才用 `s.val = 1` 过滤**，只剩约 10%。

问题在于：真正的过滤条件 `s.val = 1`（命中约 100 行）在**最后一步**才生效。前面两步 JOIN 都在围绕约 20 万行的中间结果集运作，绝大多数计算（20 万次 `t_medium` 主键查找 + 20 万次 `t_small` 主键查找）最终被丢弃，属于纯浪费。

正确的思路应是“用小结果集驱动大表”：先把 `t_small` 过滤到约 100 行，再用它驱动 `t_medium`、`t_large`，每步结果集都很小（见 good.sql）。

## MySQL 5.7 差异

5.7 与 8.0 执行计划结构一致（`STRAIGHT_JOIN` 在两版都强制顺序），差异主要在统计信息估计值上，性能特征相同：大表驱动导致中间结果集膨胀，过滤条件滞后生效。
