# EXPLAIN 参考结果 - good.sql (优化后)

## MySQL 8.0（实测 8.0.46，大表 20 万行 + 小表 5000 行）

```
+----+-------------+-------+------+---------------------+--------------+---------+-------------------------+------+----------+-------+
| id | select_type | table | type | possible_keys       | key          | key_len | ref                     | rows | filtered | Extra |
+----+-------------+-------+------+---------------------+--------------+---------+-------------------------+------+----------+-------+
|  1 | SIMPLE      | p     | ref  | idx_promotion,idx_order_no | idx_promotion | 4   | const                   |  479 |   100.00 | NULL  |
|  1 | SIMPLE      | o     | ref  | idx_order_no        | idx_order_no | 130     | sql_treasure.p.order_no |    1 |   100.00 | NULL  |
+----+-------------+-------+------+---------------------+--------------+---------+-------------------------+------+----------+-------+
```

（使用 STRAIGHT_JOIN 强制小表 t_promotion_ref 在前作为驱动表）

## 关键改进

| 字段 | 值 | 分析 |
|------|-----|------|
| 驱动表 p | `type=ref` | **小表通过 idx_promotion 精确定位** |
| 驱动表 rows | **479** | 外层循环仅 479 次（promotion_id=1 过滤后） |
| 被驱动表 o | `type=ref` | 大表通过 idx_order_no 索引查找 |
| 被驱动表 rows | **1** | 每次索引查找只扫 1 行 |
| 被驱动表 filtered | 100.00% | 无需额外过滤 |

## 为什么快

小表做驱动表：
1. 外层循环：小表经 `promotion_id=1` 索引定位，仅 479 行
2. 内层循环：每行用 `order_no` 去大表走 idx_order_no 索引，精确 1 行
3. 总查找次数：479 × 1 = **479 次**（vs bad 的 20 万次驱动循环）

## 量化对比

| 指标 | bad.sql | good.sql | 提升 |
|------|---------|----------|------|
| 驱动表 rows | 199,430 | **479** | **417 倍** |
| 驱动表 type | ALL（全表扫描） | ref（索引查找） | - |
| 耗时 | 4936 ms | **2262 ms** | **2.2 倍** |

::: tip 注意
即使两张表的 JOIN 列都有索引，驱动表选错仍然导致 2 倍以上的性能差距。如果被驱动表 JOIN 列没有索引（bad 方案的极端情况），差距会达到数千倍甚至超时。
:::
