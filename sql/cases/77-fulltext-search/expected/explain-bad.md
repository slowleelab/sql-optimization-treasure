# EXPLAIN 参考结果 - bad.sql（LIKE '%关键词%' 全表扫描）

## MySQL 8.0（20 万行中文文章）

```
+----+-------------+----------------------+------------+------+---------------+------+---------+------+--------+----------+-------------+
| id | select_type | table                | partitions | type | possible_keys | key  | key_len | ref  | rows   | filtered | Extra       |
+----+-------------+----------------------+------------+------+---------------+------+---------+------+--------+----------+-------------+
|  1 | SIMPLE      | t_article_search_bad | NULL       | ALL  | NULL          | NULL | NULL    | NULL | 199687 |    11.11 | Using where |
+----+-------------+----------------------+------------+------+---------------+------+---------+------+--------+----------+-------------+
```

## 关键问题

| 字段 | 值 | 分析 |
|------|-----|------|
| type | `ALL` | 全表扫描 |
| possible_keys | `NULL` | 前导通配符 % 使索引无法使用，无可用索引 |
| key | `NULL` | idx_category 不匹配 content 搜索条件 |
| rows | ~199,687 | 扫描全部 20 万行 |
| filtered | 11.11% | 绝大多数行被过滤掉 |
| Extra | `Using where` | 逐行检查 content 是否包含 '性能优化' |

## 为什么慢

`LIKE '%性能优化%'` 的前导通配符 `%` 决定了匹配可以出现在 content 的任意位置，B+ 树索引无法确定扫描起点（不像 `LIKE '性能优化%'` 可以定位到以该词开头的位置）。

content 字段是 `TEXT` 类型，没有 FULLTEXT 索引，优化器只能：

```
MySQL 执行流程:
1. 全表扫描 t_article_search_bad 的 20 万行                ← O(n)
2. 对每行读取 content（TEXT 可能存储在溢出页，需额外 I/O）
3. 在 content 中做子串匹配，判断是否包含 '性能优化'
4. 命中的行加入结果集
```

TEXT 字段的内容可能存储在行外溢出页，每行匹配时还需额外读取溢出页，I/O 放大严重。实际耗时约 **3.2 秒**。

::: warning 为什么 idx_category 也帮不上忙
WHERE 条件只有 `content LIKE '%...%'`，没有 category 等值条件。idx_category 索引与 content 搜索无关，无法被使用。即使加上 `category = '性能优化'` 先过滤，剩余行仍需逐行 LIKE 匹配，治标不治本。
:::

## MySQL 5.7 差异

5.7 与 8.0 行为一致，均为全表扫描，耗时相当。5.7 的 InnoDB FULLTEXT 功能同样存在，但 LIKE 写法两版都无法走索引。
