# EXPLAIN 参考结果 - bad.sql (大表实时 GROUP BY 聚合)

## MySQL 8.0（30 万行数据）

```
+----+-------------+-----------------+------------+-------+---------------+-------------+---------+------+--------+----------+----------------------------------------------+
| id | select_type | table           | partitions | type  | possible_keys | key         | key_len | ref  | rows   | filtered | Extra                                        |
+----+-------------+-----------------+------------+-------+---------------+-------------+---------+------+--------+----------+----------------------------------------------+
|  1 | SIMPLE      | t_order_report  | NULL       | range | idx_created   | idx_created | 5       | NULL | 148679 |   100.00 | Using index condition; Using temporary;      |
|    |             |                 |            |       |               |             |         |      |        |          | Using filesort                               |
+----+-------------+-----------------+------------+-------+---------------+-------------+---------+------+--------+----------+----------------------------------------------+
```

## 关键问题

| 字段 | 值 | 分析 |
|------|-----|------|
| type | `range` | 走了 idx_created 范围扫描 |
| key | `idx_created` | 用了 created_at 索引 |
| rows | ~148,679 | **预估扫描约 15 万行** |
| Extra | `Using temporary; Using filesort` | **临时表 + 文件排序** |

## 为什么慢

对 30 万行明细表做 `GROUP BY DATE(created_at)` 的实时聚合：

1. **扫描大量行**：WHERE created_at >= '2026-01-01' 命中约 15 万行（半年数据）
2. **函数转换破坏索引有序性**：`DATE(created_at)` 是函数操作，无法直接利用 idx_created 的有序性做分组
3. **Using temporary**：MySQL 需要创建临时表来存储分组聚合的中间结果
4. **Using filesort**：GROUP BY 后的 ORDER BY 需要额外的排序步骤
5. **每行都要计算**：15 万行逐行计算 DATE() 函数 + SUM/COUNT 聚合，CPU 密集

### 实时聚合 vs 预聚合

```
实时聚合 (bad):
  查询时 -> 扫描 15万行 -> 逐行 DATE() -> 临时表分组 -> SUM/COUNT -> 排序 -> 返回
  每次查询都重复全量计算

预聚合 (good):
  定时任务 -> 每天增量更新汇总表（1行/天）
  查询时 -> 直接读汇总表（< 200行）-> 返回
  计算成本前置到离线，查询零计算
```

实际耗时：约 **350 ms**（实测 MySQL 8.0.46，30 万行数据）。

## MySQL 5.7 差异

5.7 行为一致，同样产生 `Using temporary; Using filesort`。
8.0 的临时表优化略好（temptable 引擎），但根本的扫描和计算开销无法消除。
