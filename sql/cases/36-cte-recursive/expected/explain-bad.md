# EXPLAIN 参考结果 - bad.sql（多次自连接查询下属）

## MySQL 8.0（实测 8.0.46，约 10 万行员工数据，5 层树深）

查询 VP-1（level 2）下第 5 层员工（通过 3 次自连接展开 level 3->4->5），返回约 20000 行。

```
+----+-------------+-------+--------+---------------+------------+---------+-------------------------+--------+----------+-------------+
| id | select_type | table | type   | possible_keys | key        | key_len | ref                     | rows   | filtered | Extra       |
+----+-------------+-------+--------+---------------+------------+---------+-------------------------+--------+----------+-------------+
|  1 | SIMPLE      | e5    | ALL    | idx_manager   | NULL       | NULL    | NULL                    | 101056 |   100.00 | Using where |
|  1 | SIMPLE      | e4    | eq_ref | PRIMARY       | PRIMARY    | 8       | sql_treasure.e5.manager_id | 1      |   100.00 | Using where |
|  1 | SIMPLE      | e3    | eq_ref | PRIMARY       | PRIMARY    | 8       | sql_treasure.e4.manager_id | 1      |   100.00 | Using where |
|  1 | SIMPLE      | e2    | eq_ref | PRIMARY       | PRIMARY    | 8       | sql_treasure.e3.manager_id | 1      |    10.00 | Using where |
+----+-------------+-------+--------+---------------+------------+---------+-------------------------+--------+----------+-------------+
```

## 关键问题

| 字段 | 值 | 分析 |
|------|-----|------|
| table | e2, e3, e4, e5 | 4 个表别名代表 4 层连接 |
| type (e5) | **`ALL`** | **e5 全表扫描 10 万行**作为驱动表 |
| type (e4/e3/e2) | `eq_ref` | 通过主键逐层回溯找上级 |
| rows (e5) | ~101,056 | 全表扫描所有员工 |
| JOIN 层级 | 3 次 JOIN | **层数硬编码为 3 层 JOIN** |

## 为什么慢

多次自连接方案有两个核心问题：

**1. 层数硬编码**
本例写死 3 次 JOIN（覆盖 level 3->4->5）。若组织架构有 6 层，此 SQL 只能查到 level 5，漏查更深层级，必须改 SQL 加 JOIN。不同子树的深度可能不同，自连接无法自适应。

**2. 执行计划低效**
优化器选择 e5（员工表，10 万行）作为驱动表全表扫描，然后通过主键 `eq_ref` 逐层回溯 e4->e3->e2 过滤出 VP-1 的下属。这意味着：
- 先扫描全部 10 万员工
- 逐行通过 manager_id 回溯 3 层上级
- 最终只保留 VP-1 下属的 20000 行

**3. 语义局限**
自连接只能返回某一层的叶子节点（本例 level 5 员工），无法同时返回中间层（总监、经理）。若要查全层级需写多个 UNION，SQL 更加冗长。

实际耗时：约 **33 ms**（返回 20000 行 level 5 员工）。

::: warning 自连接方案的致命缺陷
自连接方案的层数被 SQL 文本"焊死"。现实中组织架构、菜单树、评论楼的深度不固定且可能很深（如 10 层）。自连接要么查不全（层数不够），要么写一长串 JOIN（可读性崩溃）。递归 CTE 才是树形遍历的正解。
:::
