# EXPLAIN 参考结果 - bad.sql（联合索引最左前缀失效）

## MySQL 8.0（30 万行数据）

```
+----+-------------+-----------------+------------+------+---------------+------+---------+------+--------+----------+-------------+
| id | select_type | table           | partitions | type | possible_keys | key  | key_len | ref  |   rows | filtered | Extra       |
+----+-------------+-----------------+------------+------+---------------+------+---------+------+--------+----------+-------------+
|  1 | SIMPLE      | t_order_latest  | NULL       | ALL  | NULL          | NULL | NULL    | NULL | 299,687 |   100.00 | Using where |
+----+-------------+-----------------+------------+------+---------------+------+---------+------+--------+----------+-------------+
```

## 关键问题

| 字段 | 值 | 分析 |
|------|-----|------|
| type | `ALL` | 全表扫描 |
| possible_keys | `NULL` | 优化器认为没有可用索引 |
| key | `NULL` | 联合索引完全未被使用 |
| rows | ~299,687 | 扫描全部 30 万行 |
| Extra | `Using where` | 逐行检查 status 和 created_at 条件 |

## 为什么慢

联合索引 `idx_user_status_created (user_id, status, created_at)` 要求查询条件满足**最左前缀原则**：必须先使用 `user_id`，才能利用后续列。

本查询 `WHERE status = 1 AND created_at > '2026-01-01'` 跳过了最左列 `user_id`，B+ 树无法从索引根部定位 `status=1` 的入口（索引是按 `user_id` 排序的，`status` 在 `user_id` 之下），导致整个联合索引失效，退化为全表扫描。

30 万行数据逐行检查 `status` 和 `created_at`，耗时约 120ms。

## MySQL 5.7 差异

5.7 与 8.0 行为一致，均为全表扫描。
