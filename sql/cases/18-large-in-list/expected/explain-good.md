# EXPLAIN 参考结果 - good.sql（临时表 JOIN 替代大 IN 列表）

## MySQL 8.0（20 万行订单 + 临时表 1000 行）

> 前置条件：需先执行 `setup-good.sql` 创建并填充 `tmp_target_users` 临时表。

```
+----+-------------+-------+------------+-------+---------------+----------+---------+-----------+------+----------+-------------+
| id | select_type | table | partitions | type  | possible_keys | key      | key_len | ref       | rows | filtered | Extra       |
+----+-------------+-------+------------+-------+---------------+----------+---------+-----------+------+----------+-------------+
|  1 | SIMPLE      | t     | NULL       | index | PRIMARY       | PRIMARY  | 8       | NULL      | 1000 |   100.00 | Using index |
|  1 | SIMPLE      | o     | NULL       | ref   | idx_user      | idx_user | 8       | t.user_id |  200 |   100.00 | NULL        |
+----+-------------+-------+------------+-------+---------------+----------+---------+-----------+------+----------+-------------+
```

## 关键改进

| 步骤 | 字段 | 值 | 分析 |
|------|------|-----|------|
| 驱动表 t | type | `index` | 临时表主键全扫描 |
| 驱动表 t | key | `PRIMARY` | 走主键索引 |
| 驱动表 t | rows | ~1,000 | 仅 1000 行，小表驱动 |
| 被驱动表 o | type | `ref` | 通过 idx_user 索引查找 |
| 被驱动表 o | key | `idx_user` | 走 user_id 索引 |
| 被驱动表 o | rows | ~200 | 每个 user_id 预估匹配约 200 行（20 万 / 1000） |

## 为什么快

改写为临时表 JOIN 后，优化器选择以 `tmp_target_users`（1000 行）为驱动表，对每个 user_id 通过 `idx_user` 索引在订单表中精确查找匹配行。这是标准的**小表驱动大表** JOIN 路径。

核心改进在于驱动方向的反转：bad 方案以 20 万行的订单表全表扫描为驱动（`type=ALL`），good 方案以 1000 行的临时表为驱动（`type=index`），通过索引将每次查找定位到约 200 行。总扫描代价从 20 万次全表行扫描降为 1000 次索引查找 + 各约 200 行的索引范围扫描。

此外，临时表方案带来工程优势：临时表可建索引、可复用、SQL 文本简洁不膨胀，执行计划稳定可控，避免了 IN 列表过大时的解析开销和优化器预估失准问题。

实际耗时：约 **30 ms**（实测 MySQL 8.0，20 万行订单 + 1000 行临时表）。

## 量化对比

| 指标 | bad.sql | good.sql | 提升 |
|------|---------|----------|------|
| 耗时 | ~220 ms | ~30 ms | **7 倍** |
| 驱动表 | t_order_in 全表 20 万行 | tmp_target_users 1000 行 | 小表驱动大表 |
| 外层 type | ALL（全表扫描） | ref（索引查找） | 走 idx_user 索引 |
| 额外开销 | 派生表物化 + 临时表 | 仅临时表（setup 预建） | 无运行时物化 |

## 5.7 vs 8.0 差异

- JOIN 方案在两个版本上执行计划一致、稳定可控
- 5.7 上 bad 方案的派生表物化开销更高，good 方案提升幅度更大
- 临时表方案避免了版本相关的优化器行为差异，是最可移植的写法
