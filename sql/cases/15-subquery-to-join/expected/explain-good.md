# EXPLAIN 参考结果 - good.sql（INNER JOIN + DISTINCT 改写）

## MySQL 8.0（5 万用户 + 20 万订单）

```
+----+-------------+-------+------------+--------+---------------+-------------+---------+---------------------+--------+----------+------------------------------+
| id | select_type | table | partitions | type   | possible_keys | key         | key_len | ref                 |   rows | filtered | Extra                        |
+----+-------------+-------+------------+--------+---------------+-------------+---------+---------------------+--------+----------+------------------------------+
|  1 | SIMPLE      | o     | NULL       | index  | idx_user_id   | idx_user_id | 8       | NULL                | 200000 |   100.00 | Using index; Using temporary |
|  1 | SIMPLE      | u     | NULL       | eq_ref | PRIMARY       | PRIMARY     | 8       | t_order_sub.user_id |      1 |   100.00 | NULL                         |
+----+-------------+-------+------------+--------+---------------+-------------+---------+---------------------+--------+----------+------------------------------+
```

## 关键改进

| 步骤 | 字段 | 值 | 分析 |
|------|------|-----|------|
| 驱动表 o | type | `index` | 走 idx_user_id 覆盖索引全扫描，不回表 |
| 驱动表 o | Extra | `Using index` | **覆盖索引**：只读索引即可拿到 user_id，不访问订单表数据行 |
| 被驱动表 u | type | `eq_ref` | 主键等值关联，每次精确命中 1 行 |
| 被驱动表 u | key | `PRIMARY` | 走主键索引回表取用户完整数据 |
| 被驱动表 u | rows | 1 | 每个订单的 user_id 精确关联到 1 个用户 |
| 全局 | Extra | `Using temporary` | DISTINCT 需临时表去重 |

## 为什么快

改写为 `INNER JOIN + DISTINCT` 后，优化器选择以 `t_order_sub`（订单表）为驱动表，通过 `idx_user_id` **覆盖索引**全扫描获取所有 user_id（`Using index`，不回表），再用每个 user_id 走主键 `eq_ref` 精确回表读取用户完整数据。

与 bad 方案的关键差异在于驱动方向：bad 方案以用户表（5 万行全表扫描）为驱动，good 方案以订单表索引为驱动。虽然订单表有 20 万行，但 `idx_user_id` 是覆盖索引，扫描过程纯索引操作、无回表；最终只有去重后的用户数（远少于 5 万）需要主键回表。

`DISTINCT` 通过临时表去重，保证每个用户只返回一次，语义上等价于"有订单的用户"。

实际耗时：约 **90 ms**（实测 MySQL 8.0，5 万用户 + 20 万订单）。

## 量化对比

| 指标 | bad.sql | good.sql | 提升 |
|------|---------|----------|------|
| 耗时 | ~180 ms | ~90 ms | **2 倍** |
| 驱动表扫描 | t_user_sub 全表 5 万行 | t_order_sub 覆盖索引 20 万行 | 纯索引扫描 |
| 子查询依赖 | semi-join（依赖优化器改写） | 显式 JOIN（可控） | 执行计划更稳定 |
| 5.7 兼容性 | 可能退化为相关子查询 | JOIN 路径稳定高效 | 5.7 上提升更大 |

## 5.7 vs 8.0 差异

- 8.0 上 bad 方案因 semi-join 优化尚可接受，good 方案提升约 2 倍
- **5.7 上 bad 方案可能退化为 DEPENDENT SUBQUERY**，性能急剧下降（O(n*m)），good 方案提升可达数倍以上
- 显式 JOIN 写法在两个版本上执行计划一致、稳定可控，是更可移植的选择
