# EXPLAIN 参考结果 - bad.sql（HAVING 中过滤 status，先分组 100 万行）

## MySQL 8.0（实测 8.0.46，100 万行数据）

```
+----+-------------+----------------+------------+-------+---------------+-------------+---------+------+--------+----------+----------------+
| id | select_type | table          | partitions | type  | possible_keys | key         | key_len | ref  | rows   | filtered | Extra          |
+----+-------------+----------------+------------+-------+---------------+-------------+---------+------+--------+----------+----------------+
|  1 | SIMPLE      | t_order_having | NULL       | index | idx_user_id   | idx_user_id | 8       | NULL | 998456 |   100.00 | Using index    |
+----+-------------+----------------+------------+-------+---------------+-------------+---------+------+--------+----------+----------------+
```

## 关键问题

| 字段 | 值 | 分析 |
|------|-----|------|
| type | `index` | 索引全扫描（走 idx_user_id 索引有序扫描） |
| key | `idx_user_id` | 使用 user_id 索引，利用索引有序性避免 filesort |
| rows | ~998,456 | 扫描全部 100 万行 |
| filtered | 100.00% | 无 WHERE 过滤，所有行都进入分组 |
| Extra | `Using index` | 覆盖索引扫描，但需扫描全部行 |

## 为什么慢

`GROUP BY user_id HAVING status = 1 AND cnt > 5` 的执行流程：

1. **无 WHERE 过滤**：`status = 1` 条件放在 HAVING 中，MySQL 无法在分组前过滤
2. **全量分组**：对全部 100 万行订单按 `user_id` 分组，计算每组的 `COUNT(*)` 和 `SUM(amount)`
3. **HAVING 过滤**：分组完成后，才过滤 `status = 1` 和 `cnt > 5`

问题在于：`status = 1` 是行级条件（非聚合条件），应该放在 WHERE 中提前过滤。status 0-3 均匀分布，status=1 只占约 25 万行（1/4）。bad 方案对 100 万行做分组，good 方案只需对 25 万行做分组，分组计算量减少 75%。

虽然 `idx_user_id` 索引让 GROUP BY 避免了 filesort（利用索引有序性），但分组聚合的计算量与行数成正比。100 万行的分组聚合耗时约 25 万行的 4 倍。

实际耗时：约 **680 ms**。

## MySQL 5.7 差异

5.7 与 8.0 行为一致，HAVING 中的非聚合条件都不会提前到 WHERE 过滤。两版都需要显式改写 SQL 才能优化。
