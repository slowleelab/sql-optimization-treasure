# EXPLAIN 参考结果 - bad.sql（IN 子查询查询"技术%"部门员工）

## MySQL 8.0（30 万员工 + 100 部门）

```
+----+-------------+--------+------------+--------+---------------+-------------+---------+---------------+--------+----------+-------------+
| id | select_type | table  | partitions | type   | possible_keys | key         | key_len | ref           |   rows | filtered | Extra       |
+----+-------------+--------+------------+--------+---------------+-------------+---------+---------------+--------+----------+-------------+
|  1 | SIMPLE      | t_emp  | NULL       | ALL    | idx_dept_id   | idx_dept_id | 8       | NULL          | 300000 |   100.00 | Using where |
|  1 | SIMPLE      | t_dept | NULL       | eq_ref | PRIMARY       | PRIMARY     | 8       | t_emp.dept_id |      1 |   100.00 | Using where |
+----+-------------+--------+------------+--------+---------------+-------------+---------+---------------+--------+----------+-------------+
```

## 关键问题

| 字段 | 值 | 分析 |
|------|-----|------|
| select_type | `SIMPLE` | 8.0 已将 IN 子查询优化为 semi-join（两表均为 SIMPLE） |
| type (t_emp) | `ALL` | 外表全表扫描 30 万行 |
| key (t_emp) | `idx_dept_id` | possible_keys 含 idx_dept_id 但未实际使用（type=ALL） |
| rows (t_emp) | ~300,000 | 全表扫描员工表 |
| type (t_dept) | `eq_ref` | 主键等值关联，每次精确 1 行 |
| Extra (t_dept) | `Using where` | 关联后还需过滤 `name LIKE '技术%'` |

## 为什么慢

MySQL 8.0 将 `IN` 子查询自动改写为 semi-join，执行计划以 `t_emp`（30 万行）为驱动表做**全表扫描**，逐行通过 `dept_id` 走 `t_dept` 主键 `eq_ref` 关联，再过滤 `name LIKE '技术%'`。

表面看 `type=eq_ref` 每次只查 1 行很高效，但问题在于驱动表是 30 万行的大表全扫描。虽然 8.0 的 semi-join 优化避免了最坏情况，但 `SELECT *` 要求对每一行回表读取完整数据，30 万次全表扫描 + 回表的代价无法避免。

**重要说明**：在本案例的数据分布下（外表大、内表小），8.0 上 bad 和 good 的执行计划可能非常接近，性能差异不大。此案例的核心价值在于理解原理：当优化器未能改写时（如 5.7 或某些复杂查询场景），IN 子查询可能退化为低效执行计划，而 EXISTS 的显式相关子查询语义更稳健。

实际耗时：约 **150 ms**（实测 MySQL 8.0，30 万员工）。

## MySQL 5.7 差异

5.7 中此查询可能显示为 `select_type=DEPENDENT SUBQUERY`（未触发 semi-join 改写），即对 `t_emp` 每行执行一次子查询扫描 `t_dept`。虽然 `t_dept` 仅 100 行、主键查找很快，但 30 万次子查询调用的开销仍高于 8.0 的 semi-join。在更复杂的查询（多表关联、UNION 等）中，5.7 更容易出现优化器无法改写 IN 子查询的情况，此时手写 EXISTS 更可靠。
